What is a Django Template?

Foundation

A Django template is a text file (usually HTML) that contains both static HTML and special template syntax — placeholders, tags, and filters that Django replaces with real data before sending the page to the browser.

Think of it this way: a template is like a letter stencil. The stencil has the fixed design, but you fill in the blanks (name, address, date) each time you use it. Django fills in your blanks from Python data.

🔑 Key Idea

Templates run 100% server-side. The browser never sees {% block %} or {{ variable }} — it only receives the final, fully-rendered HTML. This is completely different from React/Vue which render in the browser.

Why use templates instead of just writing HTML in views?

❌ Without templates (bad)

HTML is built as strings in Python. No separation of concerns, unmaintainable, hard to read, no reuse.

✅ With templates (good)

HTML designers can edit templates. Python devs handle logic in views. Each does what they're good at.

Template Syntax — The Three Delimiters

Foundation

Django templates use three types of special syntax. Each uses different brackets and does a different job. You must know these cold — they appear in every Django project.

syntax_overview.htmlDjango Template
{# ════════════════════════════════════════════════════════ #}
{# 1.  {{ }}  — VARIABLE OUTPUT                            #}
{#     Outputs the value of a variable from the context.   #}
{#     Django auto-escapes HTML for security.              #}
{# ════════════════════════════════════════════════════════ #}

<p>Hello, {{ user.name }}!</p>           {# → Hello, Alby! #}
<p>Price: ${{ product.price }}</p>      {# → Price: $29.99 #}
<p>Items: {{ cart|length }}</p>        {# → Items: 3  (using a filter) #}


{# ════════════════════════════════════════════════════════ #}
{# 2.  {% %}  — TEMPLATE TAGS                             #}
{#     Logic, control flow, block definitions.             #}
{#     These DO things — they don't just output values.   #}
{# ════════════════════════════════════════════════════════ #}

{% extends "base.html" %}          {# declare parent template #}
{% block content %}                 {# open a named block #}
  {% if user.is_authenticated %}     {# conditional #}
    <p>Welcome back!</p>
  {% endif %}
  {% for item in cart %}           {# loop #}
    <li>{{ item.name }}</li>
  {% endfor %}
{% endblock content %}


{# ════════════════════════════════════════════════════════ #}
{# 3.  {# #}  — COMMENTS                                  #}
{#     Removed entirely before the page is sent.           #}
{#     Unlike HTML comments, these are INVISIBLE in source #}
{# ════════════════════════════════════════════════════════ #}

{# This line is completely removed from output #}
<!-- This HTML comment IS visible in page source -->
{{ }} Variable

Outputs a value. Surrounded by double curly braces. Auto-HTML-escaped for XSS safety.

{% %} Tag

Performs logic: loops, conditions, block definitions, URL resolution, form tokens.

{# #} Comment

Project Setup — Where Templates Live

Setup

Django needs to know where to find your template files. There are two conventions: a project-level templates/ folder (recommended), or per-app app/templates/ folders. Both can be used together.

Project File TreeFilesystem
myproject/
│
├── myproject/
│   ├── settings.py           # ← Configure TEMPLATES here
│   ├── urls.py
│   └── wsgi.py
│
├── myapp/
│   ├── views.py
│   ├── models.py
│   └── templates/            # ← app-level templates (APP_DIRS: True)
│       └── myapp/
│           └── post_list.html
│
├── templates/                # ← project-level templates (DIRS setting)
│   ├── base.html             # master layout
│   ├── partials/
│   │   ├── navbar.html       # included fragments
│   │   └── footer.html
│   ├── home.html
│   ├── about.html
│   └── 404.html
│
└── static/                   # CSS, JS, images ({% static %} tag)
    ├── css/style.css
    └── js/app.js
settings.pyPython
# settings.py — TEMPLATES configuration
# This is the single most important settings block for templates.

import os
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent

TEMPLATES = [
    {
        # ① Which template engine to use (always Django's built-in)
        'BACKEND': 'django.template.backends.django.DjangoTemplates',

        # ② DIRS — explicit folders Django always checks
        #    BASE_DIR / 'templates' → your project-level templates/ folder
        'DIRS': [BASE_DIR / 'templates'],

        # ③ APP_DIRS — also search app/templates/ in each installed app
        #    Set True to enable per-app template folders
        'APP_DIRS': True,

        'OPTIONS': {
            'context_processors': [
                # ④ These inject variables into EVERY template automatically
                'django.template.context_processors.debug',
                'django.template.context_processors.request',   # → request
                'django.contrib.auth.context_processors.auth',    # → user
                'django.contrib.messages.context_processors.messages', # → messages
            ],
        },
    },
]
🔍 Template Search Order

Django searches for templates in this order: (1) Each directory listed in DIRS, in order. (2) Each installed app's templates/ subfolder (when APP_DIRS: True). The first match wins. If not found: TemplateDoesNotExist error.

Creating base.html — The Master Layout

Core

base.html is the skeleton of your entire site. It contains the full HTML structure — <!DOCTYPE>, <head>, navigation, footer — with named holes (blocks) that child pages fill in. You write this once and every page inherits it.

templates/base.htmlDjango Template
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  {# ─── BLOCK: title ────────────────────────────────────────── #}
  {# Child pages override this to set the browser tab title.    #}
  {# Content between the tags is the DEFAULT (used if no child  #}
  {# overrides it).                                             #}
  <title>{% block title %}My Website{% endblock title %}</title>

  {# ─── BLOCK: extra_head ───────────────────────────────────── #}
  {# Child pages can inject page-specific <meta>, <link>, or   #}
  {# <style> tags here without touching base.html               #}
  {% block extra_head %}{% endblock extra_head %}

  {# ─── STATIC: load the static tag library first ──────────── #}
  {% load static %}
  <link rel="stylesheet" href="{% static 'css/style.css' %}">
</head>

<body class="{% block body_class %}{% endblock %}">

  {# ─── INCLUDE: navbar partial ─────────────────────────────── #}
  {# Injects the navbar.html fragment at this exact location.   #}
  {# The navbar is a separate file for easy maintenance.        #}
  {% include "partials/navbar.html" %}

  {# ─── BLOCK: messages ─────────────────────────────────────── #}
  {# Django's flash messages (shown after redirect)             #}
  {% if messages %}
    <div class="messages">
      {% for message in messages %}
        <div class="alert alert-{{ message.tags }}">{{ message }}</div>
      {% endfor %}
    </div>
  {% endif %}

  {# ─── BLOCK: content ──────────────────────────────────────── #}
  {# THE MAIN CONTENT BLOCK. Every child page overrides this.  #}
  {# This is where each page's unique HTML goes.               #}
  <main>
    {% block content %}
      <p>No content defined for this page.</p>
    {% endblock content %}
  </main>

  {# ─── INCLUDE: footer partial ─────────────────────────────── #}
  {% include "partials/footer.html" %}

  {# ─── BLOCK: extra_scripts ────────────────────────────────── #}
  {# Child pages add page-specific <script> tags here.         #}
  {# Placed at bottom of <body> for performance.               #}
  {% block extra_scripts %}{% endblock extra_scripts %}

</body>
</html>

{% extends %} — Inheriting a Parent Template

Inheritance

The {% extends %} tag tells Django: "Use this file as my skeleton. I'm going to fill in only the parts I need to change." It is the foundation of template inheritance and the single most important tag you'll use.

⚠️ Iron Rule

{% extends %} MUST be the very first tag in a child template. Nothing — not even whitespace, HTML, or other tags — can come before it. Anything written outside a {% block %} in a child template is silently ignored.

templates/home.htmlDjango Template
{# ══════════════════════════════════════════════════════════ #}
{# STEP 1: Declare the parent template.                      #}
{# This MUST be the absolute first thing in the file.        #}
{# After this line, Django loads base.html as the skeleton.  #}
{# ══════════════════════════════════════════════════════════ #}
{% extends "base.html" %}

{# ══════════════════════════════════════════════════════════ #}
{# STEP 2: Override only the blocks you need to change.      #}
{# Blocks not overridden use the parent's default content.   #}
{# ══════════════════════════════════════════════════════════ #}
{% block title %}Home — My Website{% endblock title %}

{% block content %}
<div class="hero">
  <h1>Welcome to My Website</h1>
  <p>Hello, {{ user.username }}!</p>
</div>
{% endblock content %}

{# ─── This HTML is OUTSIDE a block → silently IGNORED ─────── #}
<p>This will never appear in the rendered page.</p>

Dynamic extends — the parent can be a variable

views.py + templatePython + Django
# views.py — choose parent template based on logic
def my_view(request):
    context = {
        # Switch the parent template dynamically
        'parent_template': 'admin_base.html' if request.user.is_staff else 'base.html'
    }
    return render(request, 'my_page.html', context)

---

{# my_page.html — extends a variable #}
{% extends parent_template %}   {# ← uses the variable, not a hardcoded string #}

{% block content %}
  <p>Content here...</p>
{% endblock %}
✅ What happens when you extends

1. Django loads the parent template. 2. Parses all its block tags. 3. For each block in the child, replaces the parent's version. 4. Unoverridden blocks keep the parent's default. 5. Renders the complete merged HTML.

❌ Common Mistakes

• Putting HTML before {% extends %}
• Trying to extend two parents
• Writing content outside blocks (it's ignored)
• Forgetting to name blocks the same as in parent

{% block %} / {% endblock %} — Named Slots

Inheritance

A block is a named region of the template that child templates can override. In the parent, blocks define slots with optional default content. In the child, blocks fill those slots with custom HTML.

base.html (parent) vs child.htmlDjango Template
{# ─── IN THE PARENT (base.html) ─────────────────────────── #}
{# Defines the SLOT. Content between tags = DEFAULT.          #}
{# If no child overrides this block, "Default Sidebar" shows. #}

<aside>
  {% block sidebar %}
    <p>Default Sidebar</p>
  {% endblock sidebar %}
</aside>


{# ─── IN CHILD A — OVERRIDES the slot ───────────────────── #}
{% block sidebar %}
  <nav>
    <a href="/">Home</a>
    <a href="/about/">About</a>
  </nav>
{% endblock sidebar %}


{# ─── IN CHILD B — DOESN'T define sidebar block ─────────── #}
{# Result: "Default Sidebar" from parent is used             #}
{% block content %}
  <p>Only main content, sidebar uses parent default.</p>
{% endblock content %}

Rules for blocks

block_rules.htmlDjango Template
{# RULE 1: Block names must be UNIQUE within a single template #}
{% block content %}...{% endblock %}   {# ✅ OK #}
{% block content %}...{% endblock %}   {# ❌ ERROR: duplicate block name #}


{# RULE 2: Naming the endblock is optional but STRONGLY recommended #}
{% block content %}
  ...lots of HTML...
{% endblock %}           {# valid but hard to read in big files #}

{% block content %}
  ...lots of HTML...
{% endblock content %}   {# ✅ preferred — clear which block closes #}


{# RULE 3: Blocks CAN be nested (parent holds child block) #}
{% block content %}
  <div class="wrapper">
    {% block inner_content %}
      <p>Inner default</p>
    {% endblock inner_content %}
  </div>
{% endblock content %}


{# RULE 4: A block can have NO default content (empty slot) #}
{% block extra_head %}{% endblock extra_head %}  {# empty = optional slot #}

{{ block.super }} — Keep AND Add

Inheritance

Normally overriding a block completely replaces the parent's content. But what if you want to keep the parent's content AND add your own? Use {{ block.super }} — it inserts the parent's block content at that position.

block_super_demo.htmlDjango Template
{# ─── base.html defines shared scripts ──────────────────── #}
{% block extra_scripts %}
  <script src="/static/js/base.js"></script>
  <script src="/static/js/analytics.js"></script>
{% endblock extra_scripts %}


{# ─── Without block.super — REPLACES parent content ─────── #}
{% block extra_scripts %}
  <script src="/static/js/chart.js"></script>
  {# base.js and analytics.js are GONE — not loaded! ❌ #}
{% endblock extra_scripts %}


{# ─── With block.super — KEEPS parent AND adds ───────────── #}
{% block extra_scripts %}
  {{ block.super }}          {# ← inserts base.js + analytics.js here #}
  <script src="/static/js/chart.js"></script>  {# then add chart.js ✅ #}
{% endblock extra_scripts %}

{# RESULT: all three scripts load in the right order #}


{# ─── block.super in CSS blocks ─────────────────────────── #}
{# base.html has shared styles: #}
{% block extra_styles %}
  <link rel="stylesheet" href="/static/css/base.css">
{% endblock %}

{# dashboard.html adds dashboard-specific styles too: #}
{% block extra_styles %}
  {{ block.super }}                  {# keeps base.css #}
  <link rel="stylesheet" href="/static/css/dashboard.css">
{% endblock %}
💡 When to use block.super

Use it whenever the parent has content you still need: shared scripts, CSS links, breadcrumbs, accessibility attributes, or meta tags. Without it, you'd have to duplicate the parent's content in every child — defeating the purpose of inheritance.

{% include %} — Reusable Partials

Composition

While {% extends %} is a parent-child relationship, {% include %} is simple copy-paste injection. It loads another template file and inserts its rendered HTML at that exact location. Perfect for reusable components: navbars, footers, cards, alerts, modals, form widgets.

📌 extends vs include — Key Difference

{% extends %} = parent-child inheritance. The child IS the page. | {% include %} = fragment injection. The included file is a PIECE of the page. Included files should NOT have {% extends %} themselves.

all include variantsDjango Template
{# ─── 1. BASIC include ───────────────────────────────────── #}
{% include "partials/navbar.html" %}
{# Pastes navbar.html rendered HTML here.                    #}
{# Included file gets the FULL current template context.     #}


{# ─── 2. include WITH extra variables ────────────────────── #}
{% include "partials/alert.html" with message="Saved!" level="success" %}
{# Passes extra variables INTO the included template.        #}
{# alert.html can now use {{ message }} and {{ level }}.     #}


{# ─── 3. include with ONLY (isolated context) ────────────── #}
{% include "partials/card.html" with title="Hello" only %}
{# 'only' means: give card.html ONLY title=Hello.           #}
{# The rest of the view's context is NOT passed.            #}
{# Use for safety — prevents accidental variable leakage.   #}


{# ─── 4. DYNAMIC include (variable template name) ────────── #}
{% include template_name %}
{# template_name is a variable from the context dict:       #}
{# context = {'template_name': 'partials/premium_card.html'} #}


{# ─── 5. include INSIDE a for loop ──────────────────────── #}
{% for post in posts %}
  {% include "partials/post_card.html" with post=post %}
{% endfor %}
{# Each iteration pastes a fresh post_card.html with the    #}
{# current post object. The most common include pattern!    #}


{# ─── 6. Conditional include ─────────────────────────────── #}
{% if user.is_authenticated %}
  {% include "partials/user_menu.html" %}
{% else %}
  {% include "partials/login_button.html" %}
{% endif %}

Writing a partial file (navbar.html)

templates/partials/navbar.htmlDjango Template
{# partials/navbar.html                                      #}
{# ─── NO {% extends %} tag here! Partials are fragments.   #}
{# ─── They DO have access to the view's full context.      #}
{# ─── 'request' is available via the request context       #}
{#     processor (enabled in settings.py).                  #}

<nav class="site-nav">
  <a href="{% url 'home' %}" class="brand">MySite</a>

  {# Highlight the active link using the request path #}
  <a href="{% url 'home' %}"
     class="{% if request.resolver_match.url_name == 'home' %}active{% endif %}">
    Home
  </a>
  <a href="{% url 'about' %}"
     class="{% if request.resolver_match.url_name == 'about' %}active{% endif %}">
    About
  </a>

  {# Show different nav items based on login state #}
  {% if user.is_authenticated %}
    <a href="{% url 'logout' %}">Logout ({{ user.username }})</a>
  {% else %}
    <a href="{% url 'login' %}">Login</a>
  {% endif %}
</nav>

Views & render() — Connecting Python to HTML

View ↔ Template

The view function is the bridge between your Python logic and your templates. It fetches data, processes it, then calls render() to combine the template with a context dictionary and produce the final HTML response.

views.py — complete examplePython
# views.py
from django.shortcuts import render, get_object_or_404, redirect
from django.http import HttpResponse
from .models import Article, Category

# ─── METHOD 1: render() ─────────────────────────────────────
# render(request, template_name, context=None, ...)
# This is what you'll use 95% of the time.

def article_list(request):
    articles = Article.objects.filter(published=True).order_by('-created_at')
    categories = Category.objects.all()
    featured = Article.objects.filter(featured=True).first()

    # Build the context dict — EVERY key becomes a template variable
    context = {
        'articles': articles,        # → {{ articles }} / {% for a in articles %}
        'categories': categories,   # → {{ categories }}
        'featured': featured,       # → {{ featured.title }}
        'page_title': 'Blog',       # → {{ page_title }}
        'show_sidebar': True,       # → {% if show_sidebar %}
        'article_count': articles.count(),  # → {{ article_count }}
    }

    # render() = load template + merge context + return HttpResponse
    return render(request, 'blog/article_list.html', context)


# ─── METHOD 2: render() with get_object_or_404 ──────────────
def article_detail(request, slug):
    # If article not found → automatic 404 page (no try/except needed)
    article = get_object_or_404(Article, slug=slug, published=True)
    related = Article.objects.filter(
        category=article.category
    ).exclude(pk=article.pk)[:3]

    context = {
        'article': article,
        'related': related,
        'page_title': article.title,
    }
    return render(request, 'blog/article_detail.html', context)


# ─── METHOD 3: redirect() after form POST ───────────────────
def create_article(request):
    if request.method == 'POST':
        # process form...
        article = Article.objects.create(
            title=request.POST['title'],
            content=request.POST['content'],
        )
        # Redirect to prevent double-submit on browser back
        return redirect('article_detail', slug=article.slug)

    return render(request, 'blog/create_article.html', {})

urls.py — Wiring Views to URLs

urls.pyPython
# urls.py
from django.urls import path
from . import views

app_name = 'blog'   # namespace → {% url 'blog:article_list' %}

urlpatterns = [
    path('',              views.article_list,   name='article_list'),
    path('<slug:slug>/',  views.article_detail, name='article_detail'),
    path('create/',      views.create_article, name='create_article'),
]

# In templates: {% url 'blog:article_detail' slug=article.slug %}
#              → /my-article-title/

Context Dictionary — Passing Data to Templates

View ↔ Template

The context dictionary is the data packet sent from the view to the template. Every key in the dictionary becomes a variable accessible in the template. The values can be any Python object: strings, numbers, lists, dicts, QuerySets, model instances, booleans.

context types demoPython + Django Template
# views.py — all types of context values
context = {
    # Primitives
    'title':       'Hello World',       # string → {{ title }}
    'count':       42,                  # int → {{ count }}
    'price':       9.99,                # float → {{ price }}
    'is_sale':     True,                # bool → {% if is_sale %}
    'nothing':     None,                # None → renders as empty string

    # Collections
    'tags':        ['python', 'django', 'web'],   # list → {% for tag in tags %}
    'config':      {'theme': 'dark', 'lang': 'en'},  # dict → {{ config.theme }}

    # Django QuerySets (lazy — not evaluated until iterated)
    'articles':    Article.objects.all(),          # → {% for a in articles %}
    'user_count':  User.objects.count(),           # → {{ user_count }}

    # Model instances
    'profile':     request.user.profile,           # → {{ profile.bio }}

    # Callables — Django calls them automatically!
    'now':         datetime.now,                   # → {{ now }} (no parens needed)
}

---
{# template.html — accessing all the context values #}

<h1>{{ title }}</h1>                        {# → Hello World #}
<p>Items: {{ count }}</p>                 {# → Items: 42 #}

{# Accessing dict keys — use dot notation, NOT bracket notation #}
<p>Theme: {{ config.theme }}</p>          {# → Theme: dark #}

{# Accessing model attributes — same dot notation #}

🔑 Dot Notation Resolution Order

When Django sees {{ obj.key }} it tries in order: (1) Dictionary lookup: obj["key"] (2) Attribute access: obj.key (3) List index: obj[int(key)] (4) Method call: obj.key(). First match wins. This is why {{ article.title }} works for both model instances and dicts.

{% csrf_token %} — Cross-Site Request Forgery Protection

Security

Django's built-in CSRF protection prevents malicious websites from submitting forms on behalf of your users. Every POST form requires {% csrf_token %} inside it. Without it, Django returns a 403 Forbidden error.

🔒 Mandatory — Zero Exceptions

If your form uses method="POST", you MUST include {% csrf_token %} as the first thing inside the <form> tag. This renders a hidden <input type="hidden" name="csrfmiddlewaretoken" value="..."> that Django validates on form submission.

csrf_token_usage.htmlDjango Template
{# ─── Correct usage ──────────────────────────────────────── #}
<form action="{% url 'create_article' %}" method="POST">
  {% csrf_token %}    {# ← renders as: <input type="hidden" name="csrfmiddlewaretoken" value="Kx7..."> #}

  <input type="text" name="title" placeholder="Article title">
  <textarea name="body"></textarea>
  <button type="submit">Publish</button>
</form>


{# ─── GET forms do NOT need csrf_token ───────────────────── #}
<form action="{% url 'search' %}" method="GET">
  {# No {% csrf_token %} needed for GET forms #}
  <input name="q" type="search">
  <button type="submit">Search</button>
</form>

def highlight_search(value, search_term):

    if search_term:

            search_term,
            f'<mark>{search_term}</mark>'
        )

    return value

🐛 Debug Missing Variables

In settings.py set 'string_if_invalid': '**MISSING:%s**' — missing context vars show up visibly instead of being blank. Remember to remove for production.

🔍 Django Debug Toolbar

Install django-debug-toolbar. It shows which template is rendered, all context variables, which SQL queries fired (and from which template line), and timing data.

📋 TemplateDoesNotExist

Check: (1) template name matches exactly. (2) file is in a directory listed in DIRS. (3) If app template: app is in INSTALLED_APPS and APP_DIRS: True.

📊 Inspect Context in View

Add print(context) in your view, or use {% debug %} in the template to dump all available context variables directly to the page.

from django.urls import reverse

class Tag(models.Model):
    name  = models.CharField(max_length=50, unique=True)
    slug  = models.SlugField(max_length=50, unique=True)
    color = models.CharField(max_length=7, default='#7c6fff')   # hex eg. #ff9f43

        if not self.slug:

from django.shortcuts import render, get_object_or_404, redirect
from django.contrib import messages
from .models import Article, Tag
from .forms  import ContactForm


def home(request):
    articles = Article.objects.filter(status='published')\
                              .select_related('author')\
                              .prefetch_related('tags')\
                              .order_by('-published_at')
    featured = articles.filter(featured=True).first()
    regular  = articles.exclude(pk=featured.pk) if featured else articles
    tags     = Tag.objects.all()

    context = {
        'page_title':   'Latest Articles',   # → {{ page_title }}
        'featured':     featured,             # → {% if featured %}
        'articles':     regular,              # → {% for article in articles %}
        'tags':         tags,                 # → {% for tag in tags %}
        'show_sidebar': True,                 # → {% if show_sidebar %} in base.html
    }
    return render(request, 'blog/home.html', context)


def article_detail(request, slug):
    article = get_object_or_404(
        Article.objects.select_related('author').prefetch_related('tags'),
        slug=slug, status='published'
    )
    Article.objects.filter(pk=article.pk).update(views=article.views + 1)
    related = Article.objects.filter(
        status='published', tags__in=article.tags.all()
    ).exclude(pk=article.pk).distinct()[:3]

    context = {
        'page_title':   article.title,  # → <title>Article Title | DevBlog</title>
        'article':      article,        # → {{ article.title }}, {{ article.body }}
        'related':      related,        # → {% for a in related %}

# blog/templatetags/blog_tags.py
# Load in any template with: {% load blog_tags %}

from django import template
from django.db.models import Count, Q

register = template.Library()


# ══ FILTER: reading_time ══════════════════════════════════════
#
# Usage in template:  {{ article.body|reading_time }} min read
# Result:             "5 min read"
#
# A filter takes the value on the left of | and transforms it.
# Here: body text → estimated reading minutes

@register.filter(name='reading_time')
def reading_time(body_text):
    """Estimate reading time. Average reading speed = 200 wpm."""
    if not body_text:
        return 1
    word_count = len(str(body_text).split())
    return max(1, round(word_count / 200))


# ══ SIMPLE TAG: popular_tags ══════════════════════════════════
#
# Usage:  {% popular_tags as all_tags %}
#         {% for tag in all_tags %} {{ tag.name }} ({{ tag.article_count }}) {% endfor %}
#
# 'as all_tags' stores the result in a template variable.
# Used in sidebar.html without needing the view to pass tags.
# annotate() adds article_count as an attribute on each Tag object.

@register.simple_tag
def popular_tags(limit=10):
    from blog.models import Tag
    return (
        Tag.objects
        .annotate(article_count=Count('articles',
                                      filter=Q(articles__status='published')))
        .filter(article_count__gt=0)
        .order_by('-article_count')[:limit]
    )


# ══ INCLUSION TAG: recent_articles ════════════════════════════
#
# Usage:  {% recent_articles count=3 %}
#
# Renders partials/recent_articles.html with its own context.
# The template receives {'articles': queryset}.

@register.inclusion_tag('partials/recent_articles.html')
def recent_articles(count=5):
    from blog.models import Article
    articles = Article.objects.filter(status='published')\
                              .select_related('author')\
                              .order_by('-published_at')[:count]
    return {'articles': articles}
