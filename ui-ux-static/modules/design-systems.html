Module M06 Day 8–9

Design Systems

Build once, reuse everywhere, stay consistent forever.

Learning Objectives

  • Differentiate design tokens from hardcoded style values
  • Build a Figma component with properly named variants
  • Explain the relationship between a library and the files that consume it
  • Identify when a one-off component should be promoted to the system
  • Audit a screen for components that drifted from the source library

Core Concepts

Tokens are the single source of truth

A token like color-accent-primary is referenced everywhere a color is used. Change the token once, and every button, link, and badge updates — no hunting through files for hex codes.

Components, variants, and properties

A well-built component handles every state through properties rather than duplicate components.

  • Variants — Primary / Secondary / Ghost button styles
  • Boolean properties — icon visible, disabled state
  • Instance swap — swapping an icon without breaking the component
  • Nested components — a Card built from Button + Badge instances

Concept Deep Dive

A design system means build once, reuse everywhere — consistency by default, not by everyone remembering the rules.

1. Design Tokens — One Source of Truth

What happens here:

  • Name a design decision once (color-accent-primary, radius-md)
  • Reference the token everywhere — never the raw hex code or pixel value
TokenButtonBadgeCard

Example

A button radius was hardcoded as "8px" in twelve different files. Converting it to one radius-md token meant a rebrand updated everywhere instantly.
radius-md token12 files → 1

👉 Output: Change one value, update the whole product

2. Components & Variants — Handle States Through Properties

What happens here:

  • Build variants (Primary / Ghost / Disabled) instead of duplicate components
  • Use boolean properties (icon visible, disabled state)
  • Nest components (a Card built from Button + Badge instances)

Primary

  • Filled, high emphasis

Ghost

  • Outlined, low emphasis

Disabled

  • Non-interactive

Example

One Button component with a "variant" property replaces four separate components that used to drift out of sync with each other.
1 component4 variants

👉 Output: One component that adapts, instead of five near-identical copies drifting apart

3. Figma Libraries — Distribute the System

What happens here:

  • Publish components to a shared library
  • Files "consume" the library instead of rebuilding from scratch

Key rule:

  • A component reused in 2+ unrelated flows is the system's signal to standardize it
ComponentPublishLibraryConsuming files

Example

Two designers independently built nearly identical "empty state" illustrations in separate files — a sign it belonged in the shared library.

👉 Output: Every designer pulling from the same source, automatically

4. Component Drift — The Silent Killer

What happens here:

  • A designer detaches an instance and edits it locally
  • It silently stops receiving updates from the master component

Master component

  • Updates everywhere

Detached instance

  • Silently drifts

Example

Six months later, half the buttons on the site have one radius and half have another — nobody remembers why.
Mismatched button radius

👉 Output: (The fix) Regular audits comparing live screens back to the library

The System Loop

Design systems are maintained, not finished — new components get promoted from one-off screens, and old components get deprecated as patterns change.

Simple Real-World Flow

  1. 1Find 3+ buttons on one screen with inconsistent styling
  2. 2List every unique color/spacing value used (Token audit)
  3. 3Consolidate into a short token list (Tokens)
  4. 4Rebuild as one variant-based component (Components)
  5. 5Publish to the shared library (Figma Libraries)

Key Takeaways

  • Tokens decouple decisions from hardcoded values
  • Variants beat duplicate components every time
  • A library only works if everyone actually pulls from it
  • Drift is invisible until someone audits for it — so audit regularly
