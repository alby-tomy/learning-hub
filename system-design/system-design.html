Welcome, Future Architect — What Is System Design?

Start Here

Grab a coffee, because this is the chapter everyone skips and then regrets skipping. Every other chapter in this guide — caching, microservices, load balancers, the works — is really just a detailed answer to one giant question: "how do I take a vague business idea and turn it into a system that actually works, at scale, without falling over?" That giant question has a name. It's called System Design, and by the end of this chapter you'll understand its two halves — HLD and LLD — well enough to know exactly which chapter to read next.

Here's the plain-English definition: System design is the process of planning and structuring the architecture of a software system based on user requirements. It's the bridge between "what the business wants" and "what the engineers actually build." Done well, it quietly saves you from disaster eighteen months later. Done badly, it's the reason your app falls over the moment more than ten people show up at once.

🛍️

The Online Shopping Example: Think about the last time you bought something online. Behind that one "Buy Now" click, system design decided how the user interface, the product catalog, the payment service, and the database all talk to each other — in the right order, without losing your money or your order. You never see any of it. That invisibility is the entire point: good system design feels like nothing happened at all.

Where System Design Lives in the SDLC

If you've heard of the Software Development Life Cycle (SDLC), here's the part nobody tells beginners: you cannot skip the design phase and jump straight to coding. Well — you can, plenty of bootcamp projects do exactly that, but it's the software equivalent of building a house by stacking bricks and figuring out the floor plan afterwards. System design is the phase that gives your code a backbone: it's where you decide how the system behaves when things go wrong, not just when things go right.

Added — not in the standard syllabus: Before any of this, someone has to separate Functional Requirements (what the system must do — "users can add items to a cart") from Non-Functional Requirements (how well it must do it — "the cart page must load in under 200ms for 10,000 concurrent users"). Beginners almost always design for the first and forget the second — and the second is exactly what every later chapter in this guide (scaling, caching, replication…) exists to satisfy.

Two Lenses, One System: HLD vs LLD

Here's the mental model that makes everything click: imagine you're an architect designing a building. High-Level Design (HLD) is the satellite view — where the building goes, how many floors, where the elevators and plumbing risers run. Low-Level Design (LLD) is the blueprint for one specific room — exactly where every wire, socket, and pipe goes inside it. You need both. A beautiful floor plan with no wiring detail doesn't get built; flawless room wiring with no floor plan gets built in the wrong place.

🛰️ High-Level Design (HLD)
The big-picture blueprint. Defines the major services, how they talk to each other, and the overall data flow. Answers: "What pieces exist, and how do they fit together?" Usually owned by architects and senior engineers.
🔬 Low-Level Design (LLD)
The implementation-ready detail for one piece of that blueprint. Defines classes, methods, database schemas, and exact API contracts. Answers: "How exactly does this one module work, line by line?" Usually owned by the developer about to write the code.

High-Level Design — What You Need to Know First

HLD is usually handed to people with scar tissue — engineers who've watched a system fall over in production and learned, the hard way, what "scale" actually costs. That's not to scare you off; it just means HLD leans on a checklist of prerequisite knowledge before it gets useful.

🧰
Prerequisites for HLD
  • Basic Data Structures & Algorithms
  • Databases — when to reach for SQL vs NoSQL
  • Caches (Redis, Memcached, CDNs) and what they're for
  • Functional vs Non-Functional Requirements, deeply
  • Networking — DNS, TCP/UDP, HTTP, WebSockets, TLS/SSL
  • Security basics — OAuth, JWT, rate-limiting
  • Message queues / streaming (Kafka, RabbitMQ)
  • Monoliths vs microservices, load balancing, fault tolerance
  • Observability — Prometheus, Grafana, ELK, PagerDuty
📋
What HLD Actually Covers
  • System architecture overview — services, queues, databases
  • Data flow & integrations between modules
  • Tech stack & infrastructure choices
  • Module responsibilities — what owns what
  • Trade-offs — performance, scalability, security, cost
  • Artifacts — architecture diagrams, deployment diagrams, ER overviews

HLD in the Wild — It's Not Just Theory

🎬 Netflix
Migrated their entire backend from a monolith to microservices (finishing around 2011) so the system could scale fast during holiday-season traffic spikes — a textbook HLD decision driven by a non-functional requirement: availability under load.
🚗 Uber
Runs on an event-driven architecture: a ride request, a location update, a fare change — each is an event that triggers real-time driver matching, billing, and dynamic pricing. One design decision, three downstream systems reacting independently.
🐦 Twitter
Uses a load-balanced, heavily cached architecture so trending topics and popular tweets are served from memory instead of hammering the database every time millions of people refresh their feed at once.

Low-Level Design — Zooming Into One Room

If HLD is "what services exist," LLD is "what does this service actually look like inside?" This is where the blueprint turns into class diagrams, method signatures, and database tables you can hand straight to a developer with no ambiguity left.

🧰
Prerequisites for LLD
  • Basic Data Structures & Algorithms
  • Strong grasp of OOP — encapsulation, inheritance, polymorphism, abstraction

That's it — LLD is more about disciplined thinking than a long tool list. See Chapter 23 for the full OOP/SOLID/UML deep-dive.

📋
What LLD Actually Covers
  • Component breakdown — class responsibilities, methods, attributes
  • Database schema — tables, keys, indexes, relationships
  • API & interface contracts — exact requests, responses, error codes
  • Error handling & validation per module
  • Design patterns & SOLID — for clean, extensible code
  • UML & pseudocode — class diagrams, sequence diagrams

Your 7-Step Field Guide to Designing Any System

Whether you're designing a food delivery app or a bank, the steps barely change — only the details do. Here's the walk-through, with one running example so you can see it click into place.

1
Understand Requirements
Gather and clarify business needs before drawing a single box. Example: a food delivery app needs login, restaurant listings, order placement, and online payment.
2
Define Architecture
Plan the overall structure and how components interact. Example: an e-commerce system splits into a user service, product service, order service, and payment service.
3
Choose Tech Stack
Pick technologies that fit your requirements and scale target. Example: React frontend, Node.js backend, MongoDB database.
4
Design Modules
Break the system into components with single, clear responsibilities. Example: a banking system splits into account management, transaction processing, and customer management.
5
Plan for Scalability
Design for growth before growth forces your hand. Example: a video platform uses multiple servers and caching to serve millions of concurrent viewers.
6
Ensure Security & Privacy
Bake in auth, encryption, and secure communication from day one. Example: an online banking system uses secure login, encrypted transactions, and MFA.
7
Test & Validate
Confirm the system meets functional and non-functional requirements. Example: testing a ride-hailing app to ensure booking, driver matching, and payment all hold up under load.
🧮

Added — Capacity Estimation 101 (the math beginners skip): Step 5, "Plan for Scalability," hides a skill most newcomers never practice: rough math, done on a napkin, before you design anything. Suppose your app has 10 million daily active users, each making 5 requests a day. That's 50M requests/day ÷ 86,400 seconds ≈ 580 requests per second on average — but real traffic spikes, so you design for peak (often 3-5× average, so ~2,000-3,000 RPS). If each user record is 1KB and you store 10M users, that's roughly 10GB of user data alone — tiny for a single modern DB, but the calculation habit is what matters. This "back-of-the-envelope" math is exactly what real interviewers (and real on-call engineers) expect you to do before drawing a single box. You'll see it applied properly in the URL Shortener and Chat System walkthroughs in Chapter 22.

Tips & Tricks for Cracking a System Design Problem

Here's the secret experienced engineers won't always say out loud: there is no single correct answer. Two senior engineers can design the same system completely differently and both be right, as long as they can defend their trade-offs. So the goal isn't memorizing "the" answer — it's building a repeatable process.

❓ Clarify Requirements First
Ask questions before assuming anything. Wrong assumptions compound into a wrong design fast.
✏️ Sketch the High-Level First
Clients, servers, databases, APIs — the rough shape before any detail. This is your HLD pass.
🧩 Break Into Components
Smaller services with clear responsibilities are easier to explain, test, and scale independently.
📈 Think Scale & Performance
Load balancing, caching, sharding — bring these up before someone asks "but what if it's popular?"
⚖️ Always Discuss Trade-offs
Why this database, not that one? Cost vs performance, simplicity vs scalability — say it out loud.
🔁 Refine Gradually
High-level first, then go deeper into database design, APIs, and data flow. Don't try to nail every detail in one pass.

The Engineer's Checklist — 8 Pillars to Always Consider

Whatever you design, run it past this checklist before you call it done. Miss one of these and it usually doesn't show up in week one — it shows up six months later, at 2 AM, during an incident.

📈 Scalability
Can it handle more load by scaling horizontally or vertically as needed?
⚡ Performance
Does it respond efficiently, with minimal latency?
🛡️ Reliability
Is it available with minimal downtime or failures?
🔐 Security
Does it prevent unauthorized access and protect sensitive data?
🔧 Maintainability
Is it easy to update, with clear docs and organised code?
🔗 Interoperability
Does it integrate cleanly with other systems via well-defined interfaces?
🙂 Usability
Is it intuitive for the humans who actually have to use it?
💰 Cost-Effectiveness
Does it meet requirements without overspending on infrastructure?

The Architectural Shift

Mindset

Here's a career truth nobody puts on a slide: the jump from mid-level developer to senior engineer isn't about writing more code, or even better code — it's a shift from "can you implement this ticket" to "can you decide what should exist in the first place." Mid-level developers are often brilliant inside a mature codebase with clear tickets. Hand them a blank whiteboard and a vague request like "build us something like Uber," though, and that confidence can evaporate fast.

That blank-whiteboard moment is exactly what separates senior engineers — not because they're smarter, but because they've done it enough times to have a process. They take a messy, half-formed business idea, ask the right clarifying questions, weigh trade-offs out loud, and walk away with a design that a team can actually build. That process is what the rest of this chapter — and frankly, the rest of this guide — is going to teach you to imitate.

Why Companies Value Architectural Decision-Making

🧭 Navigating Ambiguity
Seniors translate vague business needs into concrete, scalable technical architectures — without waiting for complete specs.
⏳ System Longevity
Architectural decisions directly impact long-term efficiency, maintenance, and scalability of the software — often for years.
🗄️ Data Strategy
Expert storage and data-layer choices ensure the system remains performant as user demand grows by orders of magnitude.
💼 Business Impact
Companies invest in senior talent to make high-stakes decisions that directly affect the customer experience and the bottom line.

The Data Layer: SQL vs NoSQL

Storage

Picture your single server from the last chapter starting to creak under real traffic. The first move almost every system makes is splitting it into two tiers: the web tier (the part that handles requests) and the data tier (the part that remembers things). Once they're separated, you can scale each independently — and now you face one of the most consequential decisions in the whole design: which kind of database actually fits this data? Get this wrong early and you'll be migrating live data under pressure later, which is nobody's favourite Saturday.

Relational Databases (RDBMS)

Think of a relational database as an extremely strict, extremely organised spreadsheet — everything lives in tables, rows, and columns, with rules about how tables relate to each other. It's the industry standard whenever your data has clear, well-defined relationships (a customer has orders, an order has line items). Its superpower, and the reason banks trust it with your money, is ACID compliance.

💡

ACID in plain English — Bank Transfer Example: Moving money from Account A to Account B must satisfy all four ACID properties simultaneously.

⚛️ Atomicity
The entire transfer is one indivisible unit. If the system fails after deducting from A but before adding to B, the entire transaction rolls back — money cannot "disappear."
✅ Consistency
The database moves from one valid state to another. The combined total balance of both accounts must remain the same before and after the transfer.
🔒 Isolation
If multiple transfers to Account B happen simultaneously, the transactions do not interfere or overwrite each other — each runs as if it were the only one.
💾 Durability
Once the transfer is confirmed, the record is saved permanently — even if the server crashes immediately afterward.

Non-Relational Databases (NoSQL)

Now imagine telling that strict spreadsheet to relax — to accept messy, irregular, or absolutely massive piles of data without complaining about structure. That's NoSQL's whole pitch: trade some of SQL's rigidity for speed and flexibility. It's not one technology — it's four genuinely different storage models, each built for a different shape of problem.

📄
Document Store
JSON-like Documents
Data stored as flexible documents. Ideal for content management and user profiles.
Examples: MongoDB, CouchDB
🏛️
Wide-Column Store
Massive Scale Writes
Optimised for high-volume writes at massive scale. Ideal for time-series and IoT data.
Examples: Cassandra, Cosmos DB
🕸️
Graph Store
Entities & Relationships
Focuses on nodes and the connections between them. Ideal for social networks and recommendations.
Examples: Neo4j, Amazon Neptune
⚡
Key-Value Store
Lightning-Fast RAM Storage
Simple, extremely fast storage primarily in RAM. Ideal for caching and sessions.
Examples: Redis, Memcached

Scaling Strategies

Capacity

Your app just got featured somewhere and traffic 10×'d overnight. Congratulations, and also: now what? Scaling is simply the process of growing your system's capacity to absorb that kind of demand — and you essentially have two strategies to choose from, the same way you'd handle a sudden crowd at a restaurant: build a bigger kitchen, or open more kitchens.

⬆️
Vertical Scaling
Scale Up — Stronger Machine
Adding more power (CPU, RAM) to a single existing server.
  • ⚠ Resource Ceiling — every server has a hard hardware limit
  • ⚠ No Redundancy — no backup if the server fails
  • ⚠ Single Point of Failure — entire app goes offline if it goes down
↔️
Horizontal Scaling
Scale Out — More Machines
Adding more servers to distribute the load across the pool.
  • ✓ Fault Tolerance — if one server fails, others serve traffic
  • ✓ No Hard Ceiling — add servers as demand grows
  • ✓ Superior for High Traffic — the industry-standard approach

Load Balancing Logic and Strategies

Availability

So you opened more kitchens — great. Now somebody has to stand at the door and decide which kitchen each customer walks into, or everyone piles into the first one and ignores the rest. That bouncer is the Load Balancer: it sits in front of your server pool and decides, request by request, who goes where — so no single machine ever becomes the unlucky one drowning in traffic while its neighbours sit idle.

Distribution Algorithms

01Round Robin
When: Identical server specs
Sequential rotation through the server pool. Simple and predictable.
02Least Connections
When: Variable session lengths
Routes to the server with the fewest active sessions.
03Least Response Time
When: Speed is the priority
Routes to the most responsive server. Best when servers have different capabilities.
04IP Hash
When: Client needs sticky sessions
Client IP mapped to a specific server. Ensures consistent connections for local state.
05Weighted
When: Mixed server capacities
Traffic assigned by capacity weight. Use when pool has servers of different specs (e.g., 16GB vs 64GB RAM).
06Geographical
When: Global services
Routes to the server physically closest to the user. Minimises latency at global scale.
07Consistent Hashing
When: Dynamic server pools
Hash ring maps users to servers. Minimises data redistribution when servers are added or removed.

Availability Features

❤️ Health Checks
The load balancer continuously monitors server status. If a server fails a check, it is flagged offline and removed from the rotation until it recovers — automatically, without manual intervention.

Implementation Tiers

💻 Software
Installed on existing servers. Flexible and cost-effective.
e.g., Nginx, HAProxy
🖥️ Hardware
High-performance physical appliances. Maximum throughput.
e.g., F5, Citrix
☁️ Cloud-based
Managed services with auto-scaling, integrated security and monitoring. Senior-preferred.
e.g., AWS ELB, Azure LB
06

Single Point of Failure (SPOF)

Risk

Quick gut-check question: if you unplugged any one server, database, or load balancer in your design right now, would the whole system go dark? If the answer is yes, you've just found a Single Point of Failure (SPOF) — and hunting these down before they find you is one of the least glamorous, most important jobs a senior engineer does. Nobody throws a party for "the system that didn't go down." But everybody remembers the one that did.

📉 Reliability
Leads to total downtime and direct business loss. Every minute offline has a measurable revenue cost.
⚙️ Scalability
Non-redundant components become bottlenecks as the system grows — choking throughput at the weakest link.
🎯 Security
Attackers target SPOFs (like a single load balancer) to execute successful DDoS attacks that take down the entire service.

Mitigation Strategies

1
Redundancy
Implement multiple load balancers so others automatically take over during a failure. No single node should be irreplaceable.
2
Continuous Monitoring
Track the health of every critical component in real time. Detect degradation before it becomes a full outage.
3
Self-Healing Systems
Automatically spin up new instances to replace failed ones without manual intervention — the system heals itself.

API Design and Architectural Styles

Interfaces

If your database schema is the foundation of your house, your API is the front door — it's the only part most other systems (and other teams) will ever actually touch. Think of an API as a contract: a promise about exactly what you'll accept and exactly what you'll hand back. Break that promise carelessly and you don't just have a bug — you have every client of that API breaking at once. Good API design is, quietly, just as important as good database design.

Design Principles

🔠 Consistency
Use uniform naming conventions (plural nouns) and casing (e.g., camelCase) across all endpoints.
✨ Simplicity
Designs should be intuitive enough for developers to use without constant documentation reference.
🔐 Security
Enforce authentication, authorization, and strict input validation on every endpoint.
⚡ Performance
Implement pagination, minimise payloads, and reduce round trips wherever possible.

Network and Transport Layer Protocols

Networking

Every single request your system has ever served travelled over a protocol you probably never think about — the postal system of the internet, quietly agreeing on envelopes and addresses so two strangers' computers can understand each other. You don't need to memorise an OSI model poster to design good systems, but knowing which protocol fits which job is the difference between a chat app that feels instant and one that feels like it's typing through molasses.

Application Layer

🔒 HTTPS
Uses TLS/SSL encryption to protect data in transit. The modern security standard — non-negotiable for all production systems.
🔄 WebSockets
Enables bidirectional, real-time communication (e.g., chat apps) without the overhead of repetitive request-response cycles.
📨 AMQP
An asynchronous protocol for message queuing (e.g., RabbitMQ) — producers and consumers communicate at different speeds.

Security: Authentication & Authorization

Security

Two words get tossed around interchangeably by beginners, and mixing them up is an easy interview tell: authentication and authorization. They sound similar, they often happen back-to-back in the same request, and they answer two completely different questions. Get the distinction wired into your brain now, because the rest of this chapter — and most of the security world — is built on top of it.

🪪 Authentication — Identity
Who are you? Verifies that you are who you claim to be before granting any access.
🔑 Authorization — Permissions
What can you do? Determines which resources and actions you are permitted to access.

Authentication Methods

MethodMechanismTrade-off
Basic / DigestCredentials in the HTTP header (Basic = base64; Digest = MD5)Simple but weak — not suitable alone for production
API KeysUnique strings passed in headers or query paramsRequire database lookups on every request
Session-basedStateful — server stores session IDs in RAM or RedisHarder to scale horizontally; session must be shared
JWT (Token-based)Signed, stateless JSON object — verified via cryptographic signatureNo database hit; but token revocation is complex

Access & Refresh Tokens

⚡ Access Token
Short-lived token used for API access. Expires quickly (minutes to hours) to limit exposure if compromised.
🔄 Refresh Token
Long-lived token used to renew access tokens. Security Rule: Always store in HTTP-only cookies to prevent XSS-based theft.

Authorization Models

👥
RBAC
Role-Based Access Control
Permissions assigned to roles, not individuals.
e.g., Admin, Editor, Viewer
🏷️
ABAC
Attribute-Based Access Control
Access based on attributes of user, resource, and environment.
e.g., HR department users only during work hours
📋
ACL
Access Control Lists
Specific permissions attached to a resource directly.
e.g., A shared Google Doc with per-user permissions

OAuth 2.0 & OpenID Connect

🔐 OAuth 2.0
An authorization framework for delegating access to third-party applications.

⚠ Common Pitfall: OAuth 2.0 is NOT an authentication protocol — it only delegates access, it does not verify identity.
🪪 OpenID Connect (OIDC)
The authentication layer built on top of OAuth 2.0. Provides identity tokens (ID tokens) that confirm who the user is — the correct tool for "Sign in with Google."

System Protection Techniques

Defence

Here's a slightly uncomfortable truth to internalise early: the moment your system goes live, it is under attack. Not "might be" — automated bots are scanning random IP ranges for open doors right now, this minute, regardless of whether anyone has heard of your app yet. The seven layers below aren't paranoid overengineering; they're the baseline seatbelt-and-airbags every production system needs before it's allowed near real users.

🚦 Rate Limiting
Problem
Attackers overwhelming the API with thousands of requests per minute — DDoS or brute-force login attempts.
Solution
Implement per-IP or per-user limits to block traffic automatically once a threshold is reached.
🌐 CORS
Problem
Malicious websites tricking a user's browser into making unauthorised requests to your API.
Solution
Define an allow-list of specific domains permitted to interact with your API via the CORS policy headers.
💉 Injection Prevention
Problem
Attackers inserting malicious code into input fields to read or delete database data (SQL/NoSQL Injection).
Solution
Use parameterised queries or ORM safeguards — treat all input as data, never as executable code.
🧱 Firewalls (WAF)
Problem
Malicious traffic patterns — suspicious SQL keywords, strange HTTP methods — reaching your server directly.
Solution
Use a Web Application Firewall to filter known attack patterns before they hit your infrastructure.
🔒 VPNs
Problem
Exposing sensitive internal tools or administrative APIs to the public internet unnecessarily.
Solution
Limit access to a private network, requiring users to connect via a secure encrypted tunnel first.
🛡️ CSRF Protection
Problem
A logged-in user being tricked into performing unwanted actions on a different site using their active session cookies.
Solution
Use unique CSRF tokens that must be submitted and validated with every state-changing request.
🪲 XSS Mitigation
Problem
Attackers injecting malicious scripts into pages viewed by other users (e.g., in a comment section).
Solution
Sanitise and validate all user inputs — prevent execution of injected JavaScript by escaping output correctly.
🏗️

Defence in Depth: No single protection layer is sufficient. Senior engineers layer multiple mechanisms — rate limiting + WAF + CSRF tokens + input validation — so that bypassing one layer does not compromise the entire system.

Caching Strategies
PerformanceScalability

Imagine answering the exact same trivia question for the hundredth visitor today. You wouldn't re-derive the answer from scratch each time — you'd just remember it from the first time and answer instantly. That's precisely what a cache does for your application: it's a fast, in-memory store sitting between your app and the database, remembering recent answers so it doesn't have to ask the (slow, disk-bound) database the same question over and over. Done right, this single trick can cut latency by 100× and take a huge bite out of database load.

Cache-Aside (Lazy Loading)

Application controls the cache explicitly. Most common pattern.

1App checks cache first
2Cache miss → fetch from DB
3App writes result to cache
4Future reads hit cache
✓

Only caches what's actually requested. Resilient to cache failure.

Write-Through

Every write goes to cache AND database synchronously.

1App writes to cache
2Cache immediately writes to DB
3Both stores always in sync
✓

No stale data. Higher write latency — every write hits DB.

Write-Behind (Write-Back)

Write to cache immediately, flush to DB asynchronously later.

⚡

Very fast writes. Risk of data loss if cache crashes before flush.

Read-Through

Cache itself fetches from DB on a miss — app only talks to cache.

✓

Simplifies app code. Cache vendor handles DB fallback logic.

Eviction Policies

PolicyEvictsBest For
LRU — Least Recently UsedItem not accessed longestGeneral use, hot-data workloads
LFU — Least Frequently UsedItem with fewest total accessesSkewed popularity distributions
FIFO — First In, First OutOldest inserted itemSimple queues, streaming pipelines
TTL — Time To LiveItems past their expiry timeSession data, API responses

Cache Stampede (Thundering Herd)

⚠️

Problem: A popular cache key expires. Hundreds of requests hit the database simultaneously before the first one can repopulate the cache — potentially overwhelming the DB.

Mutex Lock
Only one process fetches from DB; others wait on the lock.
Probabilistic Early Expiry
Randomly refresh before TTL expires to prevent stampedes.
Background Refresh
Serve stale data while one async worker silently refreshes.

Message Queues & Event Streaming
AsyncDecoupling

Think about how you'd leave a voicemail instead of demanding someone pick up the phone right now. You say your piece, hang up, and trust they'll get to it. That's the entire idea behind message queues and event streams: instead of Service A calling Service B directly and waiting on hold, A drops a message and walks away — B picks it up whenever it's ready, even if it was offline a moment ago. This one shift, from "talking" to "leaving messages," is what lets large systems stay decoupled instead of toppling like dominoes when one service has a bad day.

DB Indexing & Sharding
PerformanceScale

Finding a word in a 900-page book with no index means reading every page until you get lucky. An index is exactly what it sounds like — a separate, pre-sorted lookup structure that lets the database jump straight to the answer instead of scanning the whole table. Sharding solves a different problem: what happens when the book itself gets too big for one shelf? You split it into volumes, spread across multiple database nodes, so no single machine has to hold all the data or absorb all the writes alone.

Database Indexes

B-Tree Index

Default in Postgres, MySQL. Balanced tree keeps data sorted. Excellent for range queries (>, <, BETWEEN) and equality lookups.

Hash Index

Maps key → bucket via hash function. O(1) equality lookups. Cannot do range queries. Used in memory stores like Redis.

Composite Index

Index on multiple columns (user_id, created_at). Satisfies queries filtering on the leftmost prefix of the columns.

Covering Index

Index includes all columns the query needs. Allows index-only scans — the DB never touches the main table rows.

⚠️

Indexes speed up reads but slow down writes — every INSERT/UPDATE/DELETE must update each index. Over-indexing is a common performance anti-pattern.

Sharding Strategies

StrategyHow It WorksTrade-off
Range-basedShard by value range (A–M → shard 1, N–Z → shard 2)Simple, but uneven distribution ("hot shards")
Hash-basedshard = hash(key) % NEven distribution, but range queries hit all shards
Directory-basedLookup table maps key → shardFlexible, but lookup table is a bottleneck
Geo-basedRoute by user geographyLow latency, but uneven load across regions
Connection Pooling
Reuse DB connections instead of opening a new one per request. PgBouncer (Postgres) and ProxySQL (MySQL) are standard poolers. Reduces connection overhead by 10-100×.
Hotspot Problem
A shard receiving disproportionate traffic (e.g., all writes going to "user_id=1" shard). Solution: virtual nodes (consistent hashing) to rebalance without re-sharding everything.

Database Replication
DurabilityAvailability

If sharding is splitting the book into volumes, replication is photocopying the whole book and handing copies to multiple librarians. Data flows from one primary node out to one or more replica nodes — and suddenly you've bought yourself three things at once: more librarians to handle read requests, copies sitting in different geographic locations, and a backup if any single copy gets damaged.

Primary-Replica (Leader-Follower)

Replica 1
Replica 2
Replica N

Reads can be served by any replica. Writes always go to primary. Primary failure requires promoting a replica — typically automated by tools like Patroni (Postgres) or Orchestrator (MySQL).

Multi-Primary (Active-Active)
Primary 1
⇄
Primary 2

Multiple primaries accept writes simultaneously. Enables write scalability and geographic distribution. Requires conflict resolution — two primaries might update the same row concurrently.

⚠️

Conflict resolution is hard. Use only when write throughput demands it.

Replication Lag

⏱️

Replication lag: the delay between a write on the primary and it becoming visible on replicas. A user who writes data and immediately reads it from a replica might see stale data. Solutions: read-your-writes consistency (route reads back to primary for the same session), or sticky sessions (pin a user to one replica).

CAP Theorem & Consistency Models
DistributedTheory

Time for the theorem that shows up in roughly every system design interview ever conducted, so let's make sure it actually sticks. The CAP Theorem is, at its heart, a "pick two" problem — like a buffet sign that says you can only carry two of three plates at once. Across a distributed system, you can't have full Consistency, full Availability, and full Partition Tolerance all simultaneously. Something always gives.

C — Consistency

Every read receives the most recent write or an error. All nodes see the same data at the same time.

A — Availability

Every request receives a response (not an error), even if it might not be the latest data.

P — Partition Tolerance

The system continues operating even when network partitions (message loss between nodes) occur.

💡

Partition Tolerance is not optional in real distributed systems — network failures are inevitable. You're really choosing between CP (sacrifice availability during partitions) or AP (sacrifice consistency during partitions).

CP Systems — Consistent + Partition Tolerant

Return errors during network partitions rather than stale data. Examples: HBase, MongoDB (with majority concern), Zookeeper. Good for: financial transactions, inventory management.

AP Systems — Available + Partition Tolerant

Always respond, but data may be stale during partitions. Examples: Cassandra, DynamoDB, CouchDB. Good for: social feeds, DNS, product catalogs.

Consistency Models

ModelGuaranteeExample
Strong ConsistencyRead always returns latest writeGoogle Spanner, Zookeeper
Eventual ConsistencyAll nodes converge to same value eventuallyDynamoDB, Cassandra, DNS
Causal ConsistencyCausally related operations seen in order by allMongoDB (causal sessions)
Read-Your-WritesYou always see your own recent writesSticky session routing

PACELC Extension

📐

PACELC extends CAP: even when the network is not partitioned, there's a latency (L) vs consistency (C) trade-off. A system like Cassandra chooses low latency (eventual consistency). Spanner chooses consistency (higher latency). CAP only describes partition scenarios; PACELC describes the normal operating trade-off too.

Microservices Architecture
ArchitectureScale

Imagine one giant Swiss Army knife versus a toolbox full of separate, specialised tools. The Swiss Army knife is simple to carry around — until the screwdriver breaks and you have to send the whole knife in for repair. That's the monolith-versus-microservices trade-off in a nutshell. A monolith keeps all logic in one deployable unit — simple to start with. Microservices split the application into small, independently deployable services, each owning one business capability and its own data store — more moving parts, but no single broken screwdriver takes down the whole toolbox.

Monolith
  • Single deployable unit
  • Shared database
  • Simple to develop and test early on
  • Scaling means scaling everything
  • One bad deploy takes down the whole app
Microservices
  • Many small independent services
  • Each service owns its own DB
  • Scale individual services independently
  • Deploy and release independently
  • More operational complexity

Key Microservices Patterns

API Gateway

Single entry point for all clients. Handles auth, rate limiting, routing, and request aggregation. Clients never call internal services directly. Examples: Kong, AWS API Gateway, Nginx.

Circuit Breaker

Detects when a downstream service is failing and stops calling it ("opens the circuit") — returning a fallback immediately. After a timeout, it probes with limited requests and "closes" if the service recovers.

Saga Pattern

Manages distributed transactions across services. Each step publishes an event. On failure, compensating transactions undo previous steps. Two implementations: Choreography (event-driven) and Orchestration (central coordinator).

Service Mesh

Infrastructure layer that handles service-to-service communication — TLS, retries, timeouts, observability — transparently via sidecar proxies. Istio and Linkerd are common examples.

⚠️

Don't start with microservices. The operational overhead (service discovery, distributed tracing, network latency, data consistency) is high. Start with a well-structured monolith and extract services only where you've identified scaling bottlenecks.

Content Delivery Networks (CDN)
PerformanceGlobal Scale

Why should someone in Tokyo wait for a packet to physically travel to a server in Virginia and back, when a copy of that exact same file could just... live in Tokyo already? That's the whole premise of a CDN — a geographically distributed network of edge servers caching your content close to wherever your users actually are. Same content, dramatically shorter trip: latency drops from roughly 180ms down to around 5ms, just by moving the data closer to the person asking for it.

Push CDN

You push content to CDN edge nodes proactively. You control what's cached and when. Best for static assets that change infrequently (videos, software downloads, marketing images). Requires managing cache invalidation when content changes.

Pull CDN

Edge servers pull content from your origin on first request, then cache it for a TTL. Simpler to configure — just point DNS to the CDN. Best for web pages, API responses, and assets that need freshness control. TTL expiry triggers a refetch.

What CDNs Cache

Static Assets
JS bundles, CSS, images, fonts, videos — long TTLs (days/weeks)
HTML Pages
Server-rendered pages with short TTLs (minutes) for freshness
API Responses
Public GET endpoints — weather data, product listings, news feeds

CDN Benefits Beyond Caching

CapabilityHow It Helps
DDoS MitigationAbsorb and filter attack traffic at the edge before it hits origin
SSL TerminationHandle TLS handshake at edge — reduces origin CPU load
Image OptimisationResize, compress, convert format (WebP) on the fly at the edge
Geo-blockingBlock or redirect requests from specific countries at edge

Storage Systems
InfrastructureDurability

Not all "saving a file" is the same job. Booting an operating system, storing a video upload, and sharing a folder across ten office computers are three completely different storage problems wearing the same trench coat. Pick the wrong one and you'll either overpay for speed you don't need, or underpay and watch your database crawl. The three fundamental types — Block, Object, and File storage — exist precisely because one size never fits all.

Block Storage

Raw storage volumes mounted to a single server. The OS/filesystem manages the blocks. Low-latency, high-throughput.

Use for: Database files, VM disks, operating systems

Examples: AWS EBS, GCP Persistent Disk

Object Storage

Flat namespace of objects (files + metadata). Accessed via HTTP API. Infinitely scalable. Not mountable as a filesystem.

Use for: Images, videos, backups, logs, ML datasets

Examples: AWS S3, GCS, Azure Blob

File Storage (NAS)

Shared filesystem mountable by multiple servers simultaneously. Familiar directory/file hierarchy via NFS or SMB.

Use for: Shared config files, CMS assets, legacy app lift-and-shift

Examples: AWS EFS, Azure Files

Storage Tiers by Cost & Speed

TierLatencyCostUse Case
Memory (RAM)~100 nsVery highActive computation, caches
NVMe SSD~100 μsHighDatabases, hot data
HDD~10 msLowWarm data, logs
Object Store (S3)~50–200 msVery lowCold data, backups, archives
Glacier / ArchiveMinutes–hoursExtremely lowCompliance archives, cold DR
💡

RAID: Redundant Array of Independent Disks. RAID-1 mirrors data across two disks (redundancy). RAID-0 stripes data across disks (performance, no redundancy). RAID-5/6 stripes with parity (balance). For cloud-native systems, RAID is largely superseded by managed replication built into cloud storage services.

Monitoring & Observability
OperationsReliability

A system going down at 3 AM with zero clues left behind is every on-call engineer's nightmare. Observability is the practice of leaving yourself enough breadcrumbs — from the outside, without opening up the code mid-incident — to actually figure out what happened. Think of it as a doctor's three diagnostic tools: Metrics are the vital signs (heart rate, temperature), Logs are the detailed medical notes, and Traces are the full timeline of exactly where the patient went and what touched them along the way. Each answers a different question; you need all three to actually diagnose anything.

Metrics

Numeric time-series data aggregated over time. Cheapest to store, great for dashboards and alerting.

  • CPU usage, memory, request rate
  • Error rate, p50/p95/p99 latency
  • Tools: Prometheus, Datadog, CloudWatch
Logs

Timestamped records of discrete events. High cardinality, expensive to store at scale — but essential for debugging.

  • Structured JSON logs preferred over plaintext
  • Include trace ID to correlate with traces
  • Tools: ELK Stack, Loki, Splunk, CloudWatch Logs
Traces

Follow a single request as it flows across multiple services. Each hop is a span; the full journey is a trace.

  • Essential in microservices to find bottlenecks
  • Propagate trace ID via HTTP headers
  • Tools: Jaeger, Zipkin, AWS X-Ray, OpenTelemetry

SLA / SLO / SLI

TermDefinitionExample
SLI — Service Level IndicatorMetric you measureRequest success rate over 5 min
SLO — Service Level ObjectiveTarget for your SLI≥ 99.9% success rate per month
SLA — Service Level AgreementLegal contract with consequences99.9% uptime or 10% refund
Error BudgetAllowable downtime before SLO breach43.8 min/month at 99.9% SLO
🚨

Alerting Rule of Thumb: Alert on symptoms (high error rate, slow latency), not causes (high CPU). Users experience symptoms — causes are for post-incident analysis. Avoid alert fatigue by ensuring every alert is actionable.

High Availability & Disaster Recovery
ReliabilityOperations

Remember the SPOF gut-check from Chapter 06 — "what happens if I unplug this?" High Availability (HA) is the discipline of designing systems where the answer is always "nothing bad," even when individual components fail. Disaster Recovery (DR) is the next level up: not "a server died" but "an entire data centre just caught fire" — and the question becomes how fast, and with how much data lost, can you get back on your feet.

Availability Nines

SLOAnnual DowntimeMonthly Downtime
99% — "two nines"87.6 hours7.3 hours
99.9% — "three nines"8.76 hours43.8 minutes
99.99% — "four nines"52.6 minutes4.4 minutes
99.999% — "five nines"5.26 minutes26 seconds
Active-Active (Multi-Region)

Traffic routed to multiple regions simultaneously. All regions serve production traffic. Failover is instant — no "cold start". Higher cost: duplicate infrastructure in every region. Requires data synchronisation and conflict resolution.

Best for: Global services needing sub-100ms RTO

Active-Passive (Primary + Standby)

One region handles all traffic. A standby region replicates data but serves no traffic. On disaster, DNS is switched to standby. Cold standby is cheaper; warm standby boots faster.

Best for: Disaster recovery with cost constraints

RPO & RTO

RPO — Recovery Point Objective

How much data can you afford to lose? If RPO = 1 hour, you must back up at least every hour. Lower RPO = more frequent backups = higher cost.

RTO — Recovery Time Objective

How quickly must you recover? If RTO = 15 minutes, your DR plan must restore service within 15 minutes. Lower RTO = warm/hot standby = higher cost.

🔁

Chaos Engineering: Deliberately inject failures in production (Netflix's Chaos Monkey) to verify your HA systems actually work as designed. Systems you don't test regularly will fail when you need them most.

Distributed System Design Patterns
PatternsArchitecture

Every problem you'll hit in distributed systems has almost certainly already been solved by someone, somewhere, who then gave their solution a name so the rest of us don't have to reinvent it from a blank page. That's what a design pattern really is — a labelled, battle-tested shortcut. Senior engineers don't have better imaginations; they just recognise "oh, this is a Saga problem" faster, because they've seen the shape before.

CQRS — Command Query Responsibility Segregation

Separate the model used to read data from the model used to write data. Commands mutate state; queries read optimised projections. Enables independent scaling of reads and writes, and different data stores per side.

Command → Write DB
|
Query → Read DB
Event Sourcing

Instead of storing current state, store the sequence of events that led to that state. Current state is derived by replaying events. Natural audit log. Easy to rebuild projections. Pairs well with CQRS.

  • Events are immutable facts: OrderPlaced, PaymentReceived
  • Rewind to any point in time by replaying
Strangler Fig Pattern

Incrementally migrate a legacy monolith to microservices by routing individual features to new services one at a time, while the old monolith still handles everything else. The new system "strangles" the old one over time without a risky big-bang rewrite.

BFF — Backend for Frontend

Create a dedicated API layer per client type (web BFF, mobile BFF). Each BFF aggregates calls to internal services and returns a response shaped perfectly for that client — no over-fetching, no under-fetching.

Outbox Pattern

When a service writes to its DB, also write the outbound event to an outbox table in the same transaction. A separate process reads the outbox and publishes to the message queue. Solves the "dual-write" problem — guarantees the DB write and event publish are atomic.

Sidecar Pattern

Deploy a helper container alongside every service container. The sidecar handles cross-cutting concerns: logging, TLS, service discovery, health checks. The service code stays clean. Foundation of service meshes like Istio (Envoy proxy as sidecar).

Real-World System Design Examples
InterviewApplied

This is the chapter where everything in this guide finally gets to play together. These four problems — a URL shortener, a rate limiter, a chat system, a notification service — are the classics that show up again and again in senior engineering interviews, and there's no single "correct" diagram for any of them. What interviewers are actually grading is whether you run the process from Chapter 00: clarify requirements, estimate scale, propose a design, dig into the hard parts, and talk through trade-offs out loud instead of silently drawing boxes.

Design a URL Shortener (e.g., bit.ly)

Key Requirements
  • Given a long URL, return a short 7-char code
  • Redirect short URL → long URL in < 10ms
  • 100M URLs created/day, 10B redirects/day
  • Analytics: click count, geo, referrer
Core Design
  • ID generation: Base62 encode a distributed counter (Snowflake ID)
  • Write path: API → DB (store mapping) → return short code
  • Read path: Redis cache (hot URLs) → DB on miss → HTTP 301/302
  • DB: Cassandra or DynamoDB (key-value, massive scale)

Design a Rate Limiter

Algorithms
  • Token Bucket: N tokens refill at rate R. Request consumes one token. Allows bursting up to N.
  • Leaky Bucket: Requests queue, process at fixed rate. Smooths bursts completely.
  • Fixed Window: Count per time window. Simple but allows 2× burst at window boundaries.
  • Sliding Window Log: Store timestamp of every request. Precise, but high memory cost.
Implementation
  • Store counters in Redis with TTL (single-node latency ~1ms)
  • Use Redis INCR + EXPIRE atomically via Lua script
  • Return 429 Too Many Requests with Retry-After header
  • Distributed: use Redis Cluster or a sliding window approximation

Design a Chat System (e.g., WhatsApp)

Key Challenges
  • Real-time delivery (low latency)
  • Message ordering and delivery guarantees
  • Presence (online/offline status)
  • Push notifications for offline users
  • Group chats at scale
Core Design
  • Protocol: WebSocket for bi-directional real-time connection
  • Chat servers: Stateful — users connect to assigned server
  • Message store: Cassandra (wide-column, time-ordered, massive write scale)
  • Fan-out: For group chats, fan out message to all member servers via Kafka
  • Offline: Push via APNS (iOS) / FCM (Android)

Design a Notification System

Requirements
  • Support push, email, SMS, in-app
  • 10M notifications/day
  • At-least-once delivery
  • Rate limiting per user (no spam)
  • User preferences (opt-out per channel)
Core Design
  • API layer: Accepts notification request, checks user preferences
  • Queue per channel: Push queue, Email queue, SMS queue (Kafka topics)
  • Workers per channel: Consume queue, call third-party (FCM/APNS/Twilio/SendGrid)
  • Retry + DLQ: Failed deliveries retried with exponential backoff
  • Deduplication: Idempotency key prevents duplicate delivery
🎯

Interview Framework: (1) Clarify requirements — functional and non-functional. (2) Estimate scale — QPS, storage, bandwidth. (3) Propose high-level design. (4) Deep-dive on 2-3 key components. (5) Discuss trade-offs and failure modes. Rushing to draw boxes without requirements is the #1 mistake.

LLD Fundamentals — OOP, SOLID & UML
Low-Level DesignFoundations

Welcome to the microscope chapter. Everything before this point — load balancers, caching, CAP theorem — was about where things live and how they talk. This chapter is about something much smaller and, honestly, much closer to your day-to-day life as a developer: how do you write one class, one module, one function, so that it doesn't collapse into spaghetti the moment someone (probably you, in six months) has to change it? That's LLD. It's less about servers and more about discipline.

The Four Pillars of OOP — With Analogies That Actually Stick

If you've sat through an OOP lecture before, you've heard these four words. Here's the version that survives past the exam.

🔒
Encapsulation
The TV Remote, Not the Circuit Board
You press a button; you don't need to know which capacitor fires. A class should hide its internal mess behind a clean set of methods — bundle data and behaviour together, expose only what's needed.
🎭
Abstraction
Driving a Car Without Knowing the Engine
You know what the accelerator does, not how combustion works. Abstraction defines what an object does, hiding how — usually via interfaces or abstract classes.
👨‍👩‍👧
Inheritance
Inheriting Your Parents' Stuff (and Adding Your Own)
A SavingsAccount class can inherit everything a generic Account class already knows how to do, then add its own interest-calculation logic on top. Reuse without rewriting.
🦎
Polymorphism
One Command, Many Behaviours
Calling .speak() on a Dog barks; on a Cat, it meows. Same method name, different behaviour depending on the actual object — the backbone of flexible, extensible code.

SOLID — Five Rules That Keep Codebases From Rotting

SOLID isn't academic trivia — it's five hard-won rules that separate code you can extend for years from code that needs a rewrite every six months. Read each one, then ask "have I broken this rule this week?" The honest answer is usually yes, and that's fine — knowing the rule is what lets you notice.

S — Single Responsibility Principle
A class should have one reason to change. If your Order class calculates totals and sends emails and writes to the database, it has three reasons to change — and three ways to break.
O — Open/Closed Principle
Code should be open for extension, closed for modification. Add a new payment method by adding a new class, not by editing a giant if/else chain that already works.
L — Liskov Substitution Principle
A subclass should be swappable for its parent without breaking anything. If Square extends Rectangle but setting width independently breaks square-ness, you've violated LSP.
I — Interface Segregation Principle
Don't force a class to implement methods it doesn't need. A Robot forced to implement .eat() because it shares an interface with Worker is a design smell.
D — Dependency Inversion Principle
Depend on abstractions, not concrete classes. A NotificationService should depend on a MessageSender interface, not directly on EmailSender — so swapping in SMS later costs you nothing.

Classic Design Patterns Every Beginner Should Recognise

Patterns are just named solutions to problems so common that someone already solved them for you. You don't need to memorise all 23 from the original "Gang of Four" book — these six will cover most interview questions and most real code you'll touch early in your career.

🔂 Singleton
Problem
You need exactly one instance of something — a config manager, a connection pool — and accidentally creating two causes bugs.
Solution
The class controls its own instantiation and hands out the same instance every time it's requested.
🏭 Factory
Problem
Your code needs to create objects, but the exact type isn't known until runtime — e.g. PaymentMethod could be Card, UPI, or Wallet.
Solution
A factory method decides which concrete class to instantiate, so calling code never needs an ugly if/else on type.
🧱 Builder
Problem
An object needs a constructor with ten optional parameters, and most callers only care about three of them.
Solution
Build the object step by step with chained, readable calls — new PizzaBuilder().size(L).cheese().build().
🎯 Strategy
Problem
You have several interchangeable algorithms for the same job — e.g. sorting, or three different discount rules — and don't want a tangle of conditionals.
Solution
Encapsulate each algorithm behind a common interface, and swap them at runtime like interchangeable parts.
👀 Observer
Problem
When one object changes state, many other objects need to know — like a YouTube channel and its subscribers.
Solution
Subscribers register interest with the publisher; the publisher notifies all of them automatically on change. This is the pattern hiding inside every event listener you've ever written.
🔌 Adapter
Problem
You're plugging in a third-party library whose interface doesn't match what the rest of your code expects.
Solution
Wrap the mismatched interface in an adapter class that translates calls — exactly like a physical plug adapter for a foreign socket.
