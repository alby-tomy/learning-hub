Authentication vs Authorization

Concept

These two words are often confused but they answer completely different questions. Authentication (AuthN) asks "Who are you?" — it verifies identity. Authorization (AuthZ) asks "What are you allowed to do?" — it enforces permissions. Authentication always happens first.

🪪
Authentication (AuthN)
"Who are you?"
  • Verify identity via password, token, biometric
  • Sets request.user on every request
  • Result: anonymous or a known User object
  • Failing AuthN → 401 Unauthorized
School eg: Teacher enters email + password → server confirms identity
🛡️
Authorization (AuthZ)
"What can you do?"
  • Check roles, groups, object-level rules
  • Runs after AuthN confirms a user
  • Result: allowed or denied
  • Failing AuthZ → 403 Forbidden
School eg: Teacher is authenticated but can only edit their own students

How HTTP Sessions Work

Concept

HTTP is stateless — every request is independent; the server remembers nothing between them. Sessions solve this by having the server store state and giving the client a small session ID to reference that state on every subsequent request.

Step 1
POST /login
→
Server
Verify creds
→
Session DB
Create session row
→
Response
Set-Cookie: sessionid=…
↓ on every subsequent request
Step 2
GET /dashboard
→
Cookie
sessionid=abc123
→
Session DB
Lookup session
→
request.user
Teacher object
→
View
Run logic
Session lifecycle — what happens inside DjangoConcept
# 1. User submits login form
POST /login/  body: { email: "teacher@school.com", password: "pass" }

# 2. Server authenticates and creates session
#    Django stores in django_session table:
#    session_key  = "abc123def456..."   (random 32-char string)
#    session_data = base64(pickle({'_auth_user_id': 7, '_auth_user_backend': '...'}))
#    expire_date  = 2026-07-18 (SESSION_COOKIE_AGE seconds from now)

# 3. Response carries the cookie
Set-Cookie: sessionid=abc123def456; HttpOnly; Secure; SameSite=Lax; Path=/

# 4. Browser stores cookie, sends it on every request to same domain
GET /api/students/
Cookie: sessionid=abc123def456

# 5. Django middleware reads cookie, looks up session, sets request.user
#    SessionMiddleware → loads session data
#    AuthenticationMiddleware → sets request.user from session data

# 6. Logout = delete the session row
POST /logout/  → session row deleted → cookie becomes invalid
Key insight
The session ID itself carries no information — it's just a random key. All actual data (user ID, roles, etc.) lives on the server. This is why sessions are called stateful — the server must maintain state.

Cookies & Session Storage

Concept

A cookie is a key=value string the server sends in a Set-Cookie header. The browser stores it and automatically re-sends it on every request to that domain in a Cookie header. Cookie attributes control security behavior.

AttributeWhat it doesWhy it matters for auth
HttpOnlyJS cannot read the cookieBlocks XSS stealing session IDs
SecureOnly sent over HTTPSPrevents cleartext interception
SameSite=LaxNot sent on cross-site POSTMitigates CSRF attacks
Max-AgeLifetime in secondsControls session expiry
Path=/Sent for all paths on domainSession cookie must be site-wide
Django session storage backends + settingsPython
# settings.py

# ── Storage backend (where sessions live) ──────────────────────────────
SESSION_ENGINE = 'django.contrib.sessions.backends.db'        # default: DB table
# SESSION_ENGINE = 'django.contrib.sessions.backends.cache'       # Redis/Memcache (fast)
# SESSION_ENGINE = 'django.contrib.sessions.backends.cached_db'   # cache first, DB fallback
# SESSION_ENGINE = 'django.contrib.sessions.backends.file'        # files (not for production)
# SESSION_ENGINE = 'django.contrib.sessions.backends.signed_cookies' # all in cookie, no DB

# ── Security settings ─────────────────────────────────────────────────
SESSION_COOKIE_AGE      = 1209600   # 2 weeks in seconds (default)
SESSION_COOKIE_HTTPONLY = True      # JS cannot read (default True)
SESSION_COOKIE_SECURE   = True      # HTTPS only (set True in production)
SESSION_COOKIE_SAMESITE = 'Lax'    # CSRF protection
SESSION_COOKIE_NAME     = 'sessionid'  # the cookie name sent to browser
SESSION_SAVE_EVERY_REQUEST = False  # only save when modified
SESSION_EXPIRE_AT_BROWSER_CLOSE = False  # True = session cookie

# ── Inspecting sessions in Python ─────────────────────────────────────
from django.contrib.sessions.models import Session
session = Session.objects.get(session_key='abc123')
data = session.get_decoded()   # {'_auth_user_id': '7', '_auth_user_backend': '...'}

# ── What's in request.session inside a view ───────────────────────────
def my_view(request):
    request.session['last_visited'] = 'dashboard'   # write
    page = request.session.get('last_visited')        # read
    del request.session['last_visited']               # delete key
    request.session.flush()                             # delete whole session + cookie

Base64 Explained

Concept

Base64 is a binary-to-text encoding scheme. It converts any binary data into a string using only 64 printable ASCII characters: A–Z, a–z, 0–9, +, / (standard) or -, _ (URL-safe variant). It is not encryption — anyone can decode it instantly. It exists because HTTP headers and URLs can only carry printable text.

Standard Base64
  • 64 chars: A–Z, a–z, 0–9, +, /
  • Padding with = to align to 4-char groups
  • Used in: email (MIME), HTTP Basic auth header
  • Python: base64.b64encode()
Base64URL (URL-safe)
  • Same but: + → - and / → _
  • Padding = omitted (causes issues in URLs)
  • Used in: JWT tokens, OAuth, URL parameters
  • Python: base64.urlsafe_b64encode()
Base64 encoding/decoding — Python examplesPython
import base64, json

# ── Standard Base64 ────────────────────────────────────────────────────
raw = b"teacher@school.com:password123"
encoded = base64.b64encode(raw)
# b'dGVhY2hlckBzY2hvb2wuY29tOnBhc3N3b3JkMTIz'

decoded = base64.b64decode(encoded)
# b'teacher@school.com:password123'

# HTTP Basic Auth header uses this:
# Authorization: Basic dGVhY2hlckBzY2hvb2wuY29tOnBhc3N3b3JkMTIz

# ── Base64URL (JWT-safe) ───────────────────────────────────────────────
payload = {"user_id": 7, "role": "teacher", "exp": 1750000000}
json_bytes = json.dumps(payload, separators=(',',':')).encode()

encoded_url = base64.urlsafe_b64encode(json_bytes).rstrip(b'=')
# b'eyJ1c2VyX2lkIjo3LCJyb2xlIjoidGVhY2hlciIsImV4cCI6MTc1MDAwMDAwMH0'

decoded_back = base64.urlsafe_b64decode(encoded_url + b'==')
# b'{"user_id":7,"role":"teacher","exp":1750000000}'

# ── KEY POINT: Base64 is NOT secret ───────────────────────────────────
# Any JWT payload is readable by anyone who holds the token.
# Never put passwords or secrets in JWT claims.
# Security comes from the SIGNATURE, not from encoding.

# ── Why 3 bytes → 4 chars? ─────────────────────────────────────────────
# 3 bytes = 24 bits → split into four 6-bit groups → each maps to one of 64 chars
# Input length 3 → output length 4  (33% overhead)
# Input not divisible by 3 → padded with = to complete the last group
Where you see Base64 in auth
  • HTTP Basic Auth: Authorization: Basic base64("user:pass") — sent in every request header
  • JWT tokens: every part (header, payload, signature) is Base64URL encoded
  • Django session data: session_data in the DB is base64-encoded pickled Python dict
  • CSRF tokens: often Base64URL encoded random bytes

JWT Anatomy — Header · Payload · Signature

Concept

A JWT (JSON Web Token) is three Base64URL-encoded JSON objects joined by dots: header.payload.signature. The header describes the algorithm. The payload carries claims (data). The signature proves the token hasn't been tampered with. JWT is not encrypted by default — the payload is readable; the signature just makes it tamper-proof.

HEADER
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
PAYLOAD
eyJ1c2VyX2lkIjo3LCJyb2xlIjoidGVhY2hlciIsImV4cCI6MTc1MH0
.
SIGNATURE
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
Decode each part manually in PythonPython
import base64, json, hmac, hashlib

token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjo3LCJyb2xlIjoidGVhY2hlciIsImV4cCI6MTc1MH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c"

header_b64, payload_b64, sig_b64 = token.split('.')

def b64url_decode(s):
    padding = 4 - len(s) % 4
    return base64.urlsafe_b64decode(s + '=' * (padding % 4))

# ── Part 1: HEADER ────────────────────────────────────────────────────
header = json.loads(b64url_decode(header_b64))
# {"alg": "HS256", "typ": "JWT"}
# alg: HMAC-SHA256 — the signing algorithm
# typ: token type

# ── Part 2: PAYLOAD (claims) ──────────────────────────────────────────
payload = json.loads(b64url_decode(payload_b64))
# {"user_id": 7, "role": "teacher", "exp": 1750000000}
#
# Standard registered claims:
#   iss  (issuer)       — who issued the token
#   sub  (subject)      — who the token is about (user id)
#   exp  (expiration)   — Unix timestamp: reject after this time
#   iat  (issued at)    — Unix timestamp: when the token was created
#   jti  (JWT ID)       — unique ID, used for blacklisting
# Custom claims (anything you add): role, email, school_id, etc.

# ── Part 3: SIGNATURE ─────────────────────────────────────────────────
# Signature = HMAC-SHA256(base64url(header) + '.' + base64url(payload), SECRET_KEY)
SECRET = b"your-secret-key"
signing_input = (header_b64 + '.' + payload_b64).encode()
expected_sig = base64.urlsafe_b64encode(
    hmac.new(SECRET, signing_input, hashlib.sha256).digest()
).rstrip(b'=').decode()

is_valid = hmac.compare_digest(expected_sig, sig_b64)
# True → token is authentic and untampered
# False → token was modified or signed with a different key → REJECT

# ── SECURITY RULE ─────────────────────────────────────────────────────
# Payload is readable by anyone. Never put passwords, PII, or secrets in payload.
# Security guarantee: only the server with SECRET_KEY can produce a valid signature.
# If an attacker edits payload (e.g. changes role to 'admin'), signature breaks.

Access Token vs Refresh Token

Concept

Using a single long-lived token is a security risk — if stolen, the attacker has permanent access. The two-token pattern solves this: a short-lived access token for every API call, and a long-lived refresh token only used to obtain new access tokens.

PropertyAccess TokenRefresh Token
Lifetime15–60 minutes7–30 days
Sent whereEvery API request headerOnly to /token/refresh/ endpoint
Stored client-sideMemory (SPA) or localStorageHttpOnly cookie or secure storage
If stolenAttacker gets 15–60 min accessAttacker can get new access tokens
RevocationWait for expiry (or use blacklist)Delete from DB or blacklist
Complete token lifecycleHTTP flow
# ── Step 1: Login ─────────────────────────────────────────────────────
POST /api/auth/login/
Body: {"email": "teacher@school.com", "password": "secret"}

Response 200:
{
  "access":  "eyJhbGc...",   # short-lived, 60 min
  "refresh": "eyJhbGc..."    # long-lived, 7 days
}

# ── Step 2: Use access token for every API call ───────────────────────
GET /api/students/
Authorization: Bearer eyJhbGc...   # access token

Response 200: [{ ... }]

# ── Step 3: Access token expires → get a new one ──────────────────────
GET /api/students/
Authorization: Bearer eyJhbGc...   # expired access token

Response 401: {"detail": "Given token not valid for any token type"}

POST /api/auth/token/refresh/
Body: {"refresh": "eyJhbGc..."}     # long-lived refresh token

Response 200:
{
  "access":  "eyJhbGc...NEW",       # brand new access token
  "refresh": "eyJhbGc...NEW"        # new refresh token (ROTATE_REFRESH_TOKENS=True)
}

# ── Step 4: Logout — blacklist the refresh token ──────────────────────
POST /api/auth/logout/
Body: {"refresh": "eyJhbGc..."}     # revoke this refresh token

Response 205: {}
# Refresh token stored in outstanding_token + blacklisted_token tables
# Attempting to reuse it returns 401

Stateful vs Stateless Auth

Concept

The fundamental architectural difference: stateful auth (sessions) keeps user state on the server and gives the client a reference key. Stateless auth (JWT) embeds all state in the token itself — the server verifies without storing anything.

AspectSession (Stateful)JWT (Stateless)
Server storageSession record in DB or cacheNothing (or only blacklist)
Client sendsSession ID in cookieFull token in Authorization header
RevocationDelete session row — instantWait for expiry (or blacklist)
Horizontal scalingNeeds shared session store (Redis)Any server can verify — no sharing needed
PerformanceDB lookup on every requestIn-memory signature verify (fast)
Token sizeSmall (32-char session ID)Larger (~300–600 bytes)
Best forTraditional web apps, Django adminAPIs, SPAs, mobile, microservices
School usageDjango admin panel loginREST API for student/teacher app

Django Built-in Auth System

Module

Django ships a complete auth system in django.contrib.auth: User model, password hashing (PBKDF2-SHA256), authentication backends, login/logout views, and decorators. Zero configuration required — it's included by default.

Core auth functions + decoratorsPython
# ── INSTALLED_APPS (default — already there) ──────────────────────────
INSTALLED_APPS = [
    'django.contrib.auth',          # User, Group, Permission models
    'django.contrib.contenttypes',   # required by auth
    ...
]

MIDDLEWARE = [
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',  # sets request.user
    ...
]

# ── authenticate() — check credentials ────────────────────────────────
from django.contrib.auth import authenticate, login, logout

user = authenticate(request, username='ali', password='pass')
# Returns User object if credentials valid, else None

# ── login() — create session ──────────────────────────────────────────
if user is not None:
    login(request, user)
    # Creates django_session row, sets sessionid cookie

# ── logout() — destroy session ────────────────────────────────────────
logout(request)
# Deletes session row, browser cookie becomes invalid

# ── request.user — always available in views ──────────────────────────
request.user               # User object or AnonymousUser
request.user.is_authenticated  # True for real users
request.user.is_staff          # True for admin access
request.user.is_superuser      # True for full access

# ── @login_required — protect FBV ────────────────────────────────────
from django.contrib.auth.decorators import login_required, permission_required

@login_required(login_url='/login/')          # redirect if not authenticated
def dashboard(request):
    return render(request, 'dashboard.html')

@permission_required('students.view_student')  # redirect if no permission
def student_list(request): ...

# ── LoginRequiredMixin — protect CBV ──────────────────────────────────
from django.contrib.auth.mixins import LoginRequiredMixin, PermissionRequiredMixin

class DashboardView(LoginRequiredMixin, View):
    login_url = '/login/'

# ── Password hashing — what Django stores in DB ───────────────────────
# pbkdf2_sha256$390000$salt$hash
# Never store raw passwords. Use user.set_password('new') + user.save()
user.set_password('new_password')
user.check_password('candidate')   # True / False

Session Authentication in DRF

Module

SessionAuthentication reads the Django session cookie. Because it relies on cookies, DRF enforces CSRF validation on all unsafe methods (POST/PUT/PATCH/DELETE). Best for the browsable API and traditional Django views that also expose an API.

Session auth in DRF — settings + login view + CSRFPython
# settings.py
REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': [
        'rest_framework.authentication.SessionAuthentication',
        'rest_framework.authentication.BasicAuthentication',  # dev only
    ],
    'DEFAULT_PERMISSION_CLASSES': [
        'rest_framework.permissions.IsAuthenticated',
    ],
}

# urls.py — include DRF's login/logout views for browsable API
urlpatterns = [
    path('api-auth/', include('rest_framework.urls')),
    # This adds /api-auth/login/ and /api-auth/logout/ with session support
]

# ── CSRF with Session Auth ────────────────────────────────────────────
# DRF requires CSRF token for unsafe methods when SessionAuthentication is used.
# Include in HTML forms:
#   {% csrf_token %}
# Include in AJAX requests:
#   X-CSRFToken: (value from csrftoken cookie)

# ── Custom session login view ─────────────────────────────────────────
from django.contrib.auth import authenticate, login
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status
from rest_framework.permissions import AllowAny

class SessionLoginView(APIView):
    permission_classes = [AllowAny]
    authentication_classes = []        # no auth needed to log in

    def post(self, request):
        username = request.data.get('username')
        password = request.data.get('password')
        user = authenticate(request, username=username, password=password)
        if user is None:
            return Response({'error': 'Invalid credentials'}, status=401)
        login(request, user)            # creates session, sets cookie
        return Response({'message': 'Logged in', 'user': user.username})


class SessionLogoutView(APIView):
    def post(self, request):
        from django.contrib.auth import logout
        logout(request)
        return Response({'message': 'Logged out'})

Token Authentication in DRF

Module

DRF's built-in TokenAuthentication stores one opaque token per user in the database. No expiry by default. Good for simple APIs where JWT complexity isn't needed. Tokens are stateful — you can revoke by deleting the DB row.

Token auth setup + generate + usePython
# settings.py
INSTALLED_APPS += ['rest_framework.authtoken']

REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': [
        'rest_framework.authentication.TokenAuthentication',
    ]
}

# ── Generate token on login ────────────────────────────────────────────
from rest_framework.authtoken.models import Token
from rest_framework.authtoken.views import ObtainAuthToken
from rest_framework.response import Response

class LoginView(ObtainAuthToken):
    def post(self, request, *args, **kwargs):
        serializer = self.serializer_class(data=request.data,
                                           context={'request': request})
        serializer.is_valid(raise_exception=True)
        user = serializer.validated_data['user']
        token, created = Token.objects.get_or_create(user=user)
        return Response({'token': token.key, 'user_id': user.id})

# Or create token automatically on user creation via signal:
from django.db.models.signals import post_save
from django.contrib.auth.models import User
@receiver(post_save, sender=User)
def create_token(sender, instance, created, **kwargs):
    if created: Token.objects.create(user=instance)

# ── Client sends token in header ──────────────────────────────────────
# Authorization: Token 9944b09199c62bcf9418ad846dd0e4bbdfc6ee4f

# ── Revoke = delete row ────────────────────────────────────────────────
Token.objects.filter(user=request.user).delete()

# ── Difference from JWT ───────────────────────────────────────────────
# Token: opaque string, stored in DB, revocable instantly, no expiry by default
# JWT:   self-contained, no DB storage, expires automatically, harder to revoke

JWT with djangorestframework-simplejwt

Module

djangorestframework-simplejwt is the standard JWT library for DRF. It provides token obtain/refresh/verify/blacklist views and authentication classes out of the box.

Complete SimpleJWT setup — settings, urls, custom claimsPython
# pip install djangorestframework-simplejwt

# settings.py
from datetime import timedelta

INSTALLED_APPS += ['rest_framework_simplejwt',
                   'rest_framework_simplejwt.token_blacklist']  # for logout

REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': [
        'rest_framework_simplejwt.authentication.JWTAuthentication',
    ],
}

SIMPLE_JWT = {
    'ACCESS_TOKEN_LIFETIME':    timedelta(minutes=60),
    'REFRESH_TOKEN_LIFETIME':   timedelta(days=7),
    'ROTATE_REFRESH_TOKENS':    True,   # issue new refresh token on each refresh
    'BLACKLIST_AFTER_ROTATION':  True,   # old refresh token is blacklisted
    'UPDATE_LAST_LOGIN':         True,
    'ALGORITHM':                'HS256',
    'AUTH_HEADER_TYPES':        ('Bearer',),
    'AUTH_HEADER_NAME':         'HTTP_AUTHORIZATION',
    'USER_ID_FIELD':            'id',
    'USER_ID_CLAIM':            'user_id',
}

# urls.py
from rest_framework_simplejwt.views import (
    TokenObtainPairView, TokenRefreshView, TokenVerifyView, TokenBlacklistView
)
urlpatterns = [
    path('api/token/',           TokenObtainPairView.as_view()),  # POST login
    path('api/token/refresh/',    TokenRefreshView.as_view()),     # POST refresh
    path('api/token/verify/',     TokenVerifyView.as_view()),      # POST verify
    path('api/token/blacklist/',  TokenBlacklistView.as_view()),   # POST logout
]

# ── Custom claims serializer — add extra data to JWT payload ──────────
from rest_framework_simplejwt.serializers import TokenObtainPairSerializer

class CustomTokenSerializer(TokenObtainPairSerializer):
    @classmethod
    def get_token(cls, user):
        token = super().get_token(user)
        token['email']     = user.email
        token['full_name'] = user.get_full_name()
        token['is_staff']  = user.is_staff
        return token

# Wire it in settings:
SIMPLE_JWT['TOKEN_OBTAIN_SERIALIZER'] = 'myapp.serializers.CustomTokenSerializer'

# ── Read claims inside a view (without DB hit) ────────────────────────
def my_view(request):
    # request.user is set by JWTAuthentication after validating the token
    # To read raw claims from the token:
    from rest_framework_simplejwt.authentication import JWTAuthentication
    jwt_auth = JWTAuthentication()
    raw_token = jwt_auth.get_raw_token(jwt_auth.get_header(request))
    validated_token = jwt_auth.get_validated_token(raw_token)
    role = validated_token['role']   # read custom claim directly

Django Permission System

Module

Django auto-creates four permissions per model: add_modelname, change_modelname, delete_modelname, view_modelname. Assign them to users directly or through Groups. Check with user.has_perm().

Django permissions — assign, check, customPython
from django.contrib.auth.models import User, Group, Permission
from django.contrib.contenttypes.models import ContentType

# ── Auto-created permissions (after migrate) ──────────────────────────
# students.add_student
# students.change_student
# students.delete_student
# students.view_student

# ── Assign permission to a user ────────────────────────────────────────
perm = Permission.objects.get(codename='change_student')
user.user_permissions.add(perm)
user.has_perm('students.change_student')   # True

# ── Groups — assign permissions to groups, add users to groups ─────────
teachers = Group.objects.create(name='Teachers')
teachers.permissions.set([
    Permission.objects.get(codename='view_student'),
    Permission.objects.get(codename='change_student'),
    Permission.objects.get(codename='view_attendance'),
])
user.groups.add(teachers)
user.has_perm('students.view_student')    # True (via group)

# ── Custom permissions on a model ─────────────────────────────────────
class Student(models.Model):
    class Meta:
        permissions = [
            ('export_student', 'Can export student data to CSV'),
            ('transfer_student', 'Can transfer student to another class'),
        ]

# Check: user.has_perm('students.export_student')

# ── permission_required in DRF via DjangoModelPermissions ─────────────
from rest_framework.permissions import DjangoModelPermissions

class StudentViewSet(ModelViewSet):
    permission_classes = [DjangoModelPermissions]
    # GET list    → view permission
    # POST        → add permission
    # PUT/PATCH   → change permission
    # DELETE      → delete permission

Custom DRF Permission Classes

Module

Subclass BasePermission and implement has_permission (runs on every request) and/or has_object_permission (runs on detail views). Compose with | (OR) and & (AND).

Permission patterns — read-only, role-based, object-levelPython
from rest_framework.permissions import BasePermission, SAFE_METHODS

# ── Pattern 1: Read-only for anonymous, write for authenticated ────────
class ReadOnlyOrAuthenticated(BasePermission):
    def has_permission(self, request, view):
        if request.method in SAFE_METHODS: return True   # GET, HEAD, OPTIONS
        return request.user.is_authenticated

# ── Pattern 2: Role-based access ──────────────────────────────────────
class IsTeacherOrAdmin(BasePermission):
    message = 'Only teachers or admins can perform this action.'

    def has_permission(self, request, view):
        if not request.user.is_authenticated: return False
        return getattr(request.user, 'role', None) in ('teacher', 'admin')

# ── Pattern 3: Object-level — users only access their own records ──────
class IsOwner(BasePermission):
    def has_object_permission(self, request, view, obj):
        if request.method in SAFE_METHODS: return True
        if request.user.is_staff: return True
        return obj.user == request.user   # obj must have a 'user' FK

# ── Pattern 4: Compose with | and & ───────────────────────────────────
class MyView(APIView):
    permission_classes = [IsTeacherOrAdmin | IsOwner]   # OR

# ── Pattern 5: Per-action in ViewSet ──────────────────────────────────
class StudentViewSet(ModelViewSet):
    def get_permissions(self):
        if self.action in ('list', 'retrieve'):
            return [IsAuthenticated()]
        if self.action in ('create', 'update', 'partial_update'):
            return [IsAuthenticated(), IsTeacherOrAdmin()]
        if self.action == 'destroy':
            return [IsAdminUser()]
        return [IsAuthenticated()]

# ── Pattern 6: Return 403 vs 401 correctly ────────────────────────────
# Not authenticated (no user)   → 401  ← DRF handles this automatically
# Authenticated but no access   → 403  ← permission class returns False
# Customize the 403 message via: message = '...' on the class

User Model & Roles

The school system uses a custom User model extending AbstractBaseUser with a role field. Email is the login field (not username). This single model covers all user types — admin, teacher, student — and the role drives authorization everywhere.

from django.contrib.auth.models import AbstractBaseUser, BaseUserManager, PermissionsMixin
from django.db import models

class SchoolUserManager(BaseUserManager):
    def create_user(self, email, password=None, **extra):
        if not email: raise ValueError('Email required')
        user = self.model(email=self.normalize_email(email), **extra)
        user.set_password(password)
        user.save(using=self._db)
        return user

    def create_superuser(self, email, password, **extra):
        extra.setdefault('is_staff', True)
        extra.setdefault('is_superuser', True)
        extra.setdefault('role', SchoolUser.ADMIN)
        return self.create_user(email, password, **extra)


class SchoolUser(AbstractBaseUser, PermissionsMixin):
    ADMIN   = 'admin'
    TEACHER = 'teacher'
    STUDENT = 'student'
    ROLE_CHOICES = [
        (ADMIN,   'Administrator'),
        (TEACHER, 'Teacher'),
        (STUDENT, 'Student'),
    ]

    email     = models.EmailField(unique=True)
    full_name = models.CharField(max_length=100)
    role      = models.CharField(max_length=10, choices=ROLE_CHOICES, default=STUDENT)
    phone     = models.CharField(max_length=15, blank=True)
    is_active = models.BooleanField(default=True)
    is_staff  = models.BooleanField(default=False)    # can access admin panel
    created_at = models.DateTimeField(auto_now_add=True)

    objects = SchoolUserManager()

    USERNAME_FIELD  = 'email'    # login with email instead of username
    REQUIRED_FIELDS = ['full_name', 'role']

    class Meta:
        db_table = 'school_users'

    # ── Convenience properties for authorization checks ─────────────────
    @property
    def is_admin(self): return self.role == self.ADMIN
    @property
    def is_teacher(self): return self.role == self.TEACHER
    @property
    def is_student_role(self): return self.role == self.STUDENT

# ── Web login form view (session-based) ───────────────────────────────
from django.shortcuts import render, redirect
from django.contrib.auth import authenticate, login, logout
from django.contrib.auth.decorators import login_required

def school_login(request):
    if request.user.is_authenticated: return redirect('dashboard')
    error = None
    if request.method == 'POST':
        user = authenticate(request,
                            username=request.POST['email'],
                            password=request.POST['password'])
        if user:
            login(request, user)      # ← creates session row, sets cookie
            next_url = request.GET.get('next', 'dashboard')
            return redirect(next_url)
        error = 'Invalid email or password'
    return render(request, 'accounts/login.html', {'error': error})


@login_required
def school_logout(request):
    logout(request)      # ← deletes session, cookie becomes invalid
    return redirect('login')


@login_required
def dashboard(request):
    # request.user is loaded from session automatically
    context = {
        'user': request.user,
        'role': request.user.role,
        'is_admin': request.user.is_admin,
    }
    return render(request, 'dashboard.html', context)

JWT Login & Register API

Project

The REST API uses JWT. Teachers and students authenticate via POST /api/auth/login/ and receive access + refresh tokens. The access token is sent as Authorization: Bearer … on every subsequent API call. The JWT payload includes role, name, and school metadata.

accounts/serializers.py + views.py + urls.pyPython
# accounts/serializers.py
from rest_framework_simplejwt.serializers import TokenObtainPairSerializer
from rest_framework import serializers
from .models import SchoolUser

class SchoolTokenSerializer(TokenObtainPairSerializer):
    """Embeds role, name, and school info directly in JWT payload."""
    @classmethod
    def get_token(cls, user):
        token = super().get_token(user)
        # Custom claims — readable from token without a DB lookup
        token['email']     = user.email
        token['full_name'] = user.full_name
        token['role']      = user.role          # 'admin' | 'teacher' | 'student'
        token['is_staff']  = user.is_staff
        return token
    # Decoded payload example:
    # {"user_id": 7, "email": "ali@school.com", "full_name": "Ali Khan",
    #  "role": "teacher", "is_staff": false, "exp": 1750003600, "iat": 1749999999}


class RegisterSerializer(serializers.ModelSerializer):
    password  = serializers.CharField(write_only=True, min_length=8)
    password2 = serializers.CharField(write_only=True)

    class Meta:
        model  = SchoolUser
        fields = ['email', 'full_name', 'role', 'phone', 'password', 'password2']

    def validate(self, attrs):
        if attrs['password'] != attrs.pop('password2'):
            raise serializers.ValidationError({'password2': 'Passwords do not match.'})
        return attrs

    def create(self, validated_data):
        password = validated_data.pop('password')
        user = SchoolUser(**validated_data)
        user.set_password(password)   # hashes via PBKDF2-SHA256
        user.save()
        return user

class MeView(APIView):
    """Current user profile — reads from DB, not JWT."""
    permission_classes = [IsAuthenticated]

    def get(self, request):
        return Response({
            'id':        request.user.id,
            'email':     request.user.email,
            'full_name': request.user.full_name,
            'role':      request.user.role,
            'is_staff':  request.user.is_staff,
        })

    def patch(self, request):
        allowed = {'full_name', 'phone'}   # only allow safe fields
        data    = {k: v for k, v in request.data.items() if k in allowed}
        for field, value in data.items():
            setattr(request.user, field, value)
        request.user.save(update_fields=list(data.keys()))
        return Response({'message': 'Profile updated.'})

from rest_framework.permissions import BasePermission, SAFE_METHODS

class IsSchoolAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and request.user.is_admin


class IsTeacherOrAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and (
            request.user.is_admin or request.user.is_teacher
        )


class IsOwnerOrTeacherOrAdmin(BasePermission):
    """List: teachers/admins only. Detail: student sees own record."""

    def has_permission(self, request, view):
        return request.user.is_authenticated

    def has_object_permission(self, request, view, obj):
        if request.user.is_admin or request.user.is_teacher: return True
        # Student: only their own record
        return getattr(obj, 'user', None) == request.user


class IsTeacherOfClass(BasePermission):
    """Teacher can only modify students in their own classroom."""
    message = 'You can only edit students in your own classroom.'

    def has_permission(self, request, view):
        if request.method in SAFE_METHODS: return request.user.is_authenticated
        return request.user.is_authenticated and (
            request.user.is_admin or request.user.is_teacher
        )

    def has_object_permission(self, request, view, obj):
        if request.method in SAFE_METHODS: return True
        if request.user.is_admin: return True
        # Teacher: object must belong to their classroom
        classroom = getattr(obj, 'classroom', None)
        return classroom and classroom.teacher == request.user


# ── Wire permissions into ViewSets ────────────────────────────────────
class StudentViewSet(ModelViewSet):
    def get_permissions(self):
        if self.action in ('list',):
            return [IsTeacherOrAdmin()]
        if self.action == 'retrieve':
            return [IsOwnerOrTeacherOrAdmin()]
        if self.action in ('create',):
            return [IsTeacherOrAdmin()]
        if self.action in ('update', 'partial_update'):
            return [IsTeacherOfClass()]
        if self.action == 'destroy':
            return [IsSchoolAdmin()]
        return [IsAuthenticated()]

Auth security checklist
  • AUTH_USER_MODEL set before first migration — cannot change after
  • SESSION_COOKIE_SECURE = True and SESSION_COOKIE_HTTPONLY = True in production
  • CSRF_COOKIE_SECURE = True — protects session-based forms
  • Never put passwords, full PII, or secrets in JWT payload
  • ROTATE_REFRESH_TOKENS = True — makes stolen refresh tokens single-use
  • BLACKLIST_AFTER_ROTATION = True — old refresh tokens are revoked
  • Keep ACCESS_TOKEN_LIFETIME short (≤60 min) to limit stolen-token damage
  • Always use HTTPS — tokens in headers are cleartext without it
