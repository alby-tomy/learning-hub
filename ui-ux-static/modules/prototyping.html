Module M07 Day 9–10

Prototyping

Make it clickable before you make it real.

Learning Objectives

  • Connect screens into a clickable flow using Figma prototyping
  • Choose appropriate transition types for different navigation patterns
  • Apply easing curves that match the weight and purpose of an element
  • Build a smart-animate transition between two component variants
  • Identify when an interaction needs motion versus when it's noise

Core Concepts

Prototypes answer 'does it flow', not 'does it look good'

A prototype's job is to let users (or stakeholders) experience the sequence of decisions in an interface — testing navigation and flow logic well before development.

Motion has rules, not just vibes

Good motion design borrows directly from physics — objects accelerate and decelerate, they don't snap instantly.

  • Ease-out — elements entering the screen
  • Ease-in — elements leaving the screen
  • Duration — 150–300ms for most UI transitions
  • Smart Animate — interpolates matching layers between two frames

2. Transition Types — Match Motion to Meaning

What happens here:

  • Instant — snaps, no transition
  • Dissolve — crossfades
  • Smart Animate — interpolates matching layers between frames (most realistic)

Key rule:

  • Smart Animate requires layer names to match exactly between frames — a renamed layer breaks it silently

4. Knowing When NOT to Animate

What happens here:

  • Reserve motion for state changes that genuinely need it
  • Overusing transitions slows testers down and feels gimmicky

Animate

  • Genuine state change

Don't animate

  • Every single tap

Example

Adding a 400ms bounce to every single button tap made a prototype feel sluggish during testing — it was cut down to only the screens that actually changed state.
400ms bounce cut

👉 Output: Motion as a tool, not decoration

The Prototype Loop

Prototypes get rebuilt as testing reveals problems — a confusing transition sends you back to rethink the flow itself, and a flow that "feels slow" in testing may need fewer steps, not faster animation.

Simple Real-World Flow

  1. 1Take your mid/hi-fi screens and link them into a flow (Connecting Screens)
  2. 2Pick a transition type per navigation moment (Transition Types)
  3. 3Apply easing matched to entering/leaving elements (Easing)
  4. 4Cut any motion that doesn't communicate something (Knowing when not to)
  5. 5Move to Usability Testing

Key Takeaways

  • Prototypes validate flow logic, not visual polish
  • Smart Animate needs identical layer names to work
  • Good motion borrows from physics — accelerate, decelerate, don't snap
  • Not every screen needs an animation
